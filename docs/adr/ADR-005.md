# ADR-005: Tier 3a(標準3D)の導入方式 — vanilla Three.js + useEffect ライフサイクル

- **ステータス**: Accepted(制作者=HUMAN の「Tier1→Tier2→Tier3 へ GO」指示 2026-07-13 を Go とし、本 ADR の PR マージをもって発効)
- **決定者**: ARCHITECT(Claude Code オーケストレータ)起票 → 技術スパイク(別エージェント)→ 独立レビュー(別系統)+ 制作者事後監査
- **日付**: 2026-07-14(JST)
- **関連文書**: ADR-001(Tier 3a=Three.js は決定済み——本 ADR は**導入方式**を決める)、ADR-004(Pixi の useEffect ライフサイクル前例・a11y 条件)、`docs/DESIGN.md` §レンダリング戦略、`AGENTS.md` §3(C-3/C-4/C-6)

## 文脈

MVP 2 完了(公開29単元)により MVP 3(Tier 3: 一次変換3×3・回転行列と基底変換・曲面プロット・複素関数ドメインカラーリング)が対象になった。レンダラー自体は ADR-001 で Tier 3a=Three.js(既定 WebGL レンダラー)と決定済みだが、(a) React Island との統合方式、(b) headless e2e での検証可否、(c) バンドル影響、(d) キーボード操作(a11y DoD は削減対象外)の方式が未決だった。技術スパイク(2026-07-14、ソース: `docs/adr/assets/adr-005-three-spike/`)で vanilla 統合と @react-three/fiber(R3F)を実装比較した。

## 計測・検証(スパイク要旨)

| 観点 | 案A: vanilla + useEffect | 案B: R3F |
|---|---|---|
| コード量(同一デモ) | 146行 | 105行+`extend()`/型拡張の付帯コード |
| 追加依存 | `three` のみ(OrbitControls 同梱) | `three` + `@react-three/fiber` |
| バンドル(gzip 実測) | **135.1 KB**(three+OrbitControls) | +約103.5 KB(R3F ランタイム純増) |
| Astro Islands(client:visible) | 問題なし | 今回の版の組合せでは SSR/hydration 問題は再現せず |
| キーボード代替の実装難度 | 自前実装必須 | **同一**(OrbitControls はオービットのキー操作を持たず、R3F の宣言性の恩恵が及ばない) |
| headless e2e | 描画成功 | 描画成功 |

- headless Chromium(SwiftShader)は **WebGL1 のみ**だが、Three.js が自動フォールバックし、Tier 3a の標準用途(立方体・格子・矢印・OrbitControls)に WebGL2 は不要と確認。
- **e2e の非空ピクセル判定には `preserveDrawingBuffer: true` が必須**(既定 false では readPixels が常に0を返す——実装ノート)。
- 性能下見: 102 オブジェクト+OrbitControls、SwiftShader(ソフトウェア描画)でも 64〜66fps。実 GPU では十分な余裕。

## 決定

1. **Tier 3a の統合方式として vanilla Three.js(useEffect ライフサイクル管理)を採用する。R3F は採用しない**(将来、シーンの複雑化で命令型が不利になった場合の再評価候補として温存)。根拠: R3F の純増 ~103KB gzip に見合う便益がない——最大の懸念だったキーボード代替の実装難度が両案で同一であり、ライフサイクル管理は Pixi(ADR-004、`CltScene.tsx`)の既承認前例と完全同型で学習コストがほぼゼロ。
2. **Scene 配置は `src/components/scenes/three/`**(§6 予約済み)。Pixi と同じ規律: mount で init・unmount で dispose・StrictMode 二重起動ガード・**WebGL 初期化失敗時は fail-with-message**(C-3。Tier 3a は WebGL なので事実上全ブラウザで動くが防御)。`lib/math` は Three.js を import しない(行列・ベクトル演算は自前の純粋 TS——Three.js の Matrix3/Vector3 を数学モデルに使わない。描画層でのみ変換する)。
3. **カメラ操作の a11y**: OrbitControls(マウス/タッチ)に加え、**離散的なカメラ回転ボタン(方位角/仰角を15°刻み等)+視点リセット**をキーボード操作可能な DOM ボタンとして必ず併設する。観察表(DOM)が値の主担保である原則は Tier 2 と同じ(ADR-004 §5 を継承)。canvas は aria-hidden。
4. **e2e 実装規約**: Three.js Scene は `preserveDrawingBuffer: true` で生成し、非空ピクセル判定を可能にする(教材規模では性能影響は無視できる)。RTL は Scene をスタブ(Pixi 前例)。**描画幾何(射影・変換の整合)は RTL で不可視のため、レビュー観点に「座標変換の単一定義」を必須で含める**(PR #38 の教訓の Tier 3 版)。
5. **Tier 3b(WebGPU)の判断枠組み**: MVP 3 の「複素関数ドメインカラーリング」は、まず **Three.js の ShaderMaterial(WebGL フラグメントシェーダー)**で実装を試みる(ピクセル単位計算はフラグメントシェーダーの本来業務であり、WebGPU compute を要する規模ではない見込み)。WebGL1/2 のシェーダーで品質・性能が立たないと**実測で示された場合のみ** Tier 3b(WebGPU、C-3 のメッセージ表示運用)へ進む。「安易に 3D/ピクセル=WebGPU としない」(DESIGN)の具体化。
6. 依存追加: `three`(MIT)+ `@types/three`。バージョンは package.json で固定し、更新は依存審査に従う(本 ADR が審査記録を兼ねる)。バンドル +135KB gzip は Astro Islands の遅延読込により該当ページのみに局所化する(Pixi 前例で検証済みのメカニズム)。

## 検討した代替案

| 案 | 長所 | 短所 | 採否 |
|---|---|---|---|
| **vanilla Three.js + useEffect(本決定)** | 追加ランタイム依存ゼロ・Pixi 前例と同型・~103KB 軽い | 命令型のシーン構築コード | **採用** |
| @react-three/fiber | 宣言的 JSX、React 状態との自然な同期 | +103.5KB gzip。OrbitControls 等は結局 `extend()` の手作業。a11y 要件(自前キーボード代替)に宣言性の恩恵が及ばない | 不採用(将来の再評価候補) |
| Babylon.js | 統合的なエンジン機能 | ADR-001 の決定(Three.js)を覆す理由がない。バンドルも大きい | 不採用 |
| WebGPU(Three.js WebGPURenderer 含む) | 将来性 | Tier 3a の互換要件(主要モダンブラウザ全て=WebGL)に反する。ADR-001 で Tier 3b 限定と決定済み | 不採用(Tier 3b 判断は決定 §5 の枠組みで) |
| 3D を自前 SVG 投影で描く(Tier 1 拡張) | 依存追加なし | カメラ操作・奥行き・遮蔽の自前実装は Three.js 採用より高コスト。ADR-001 で退けた方向 | 不採用 |

## 結果

**利点**: Tier 2(Pixi)で確立したライフサイクル・a11y・e2e の規律をそのまま Tier 3a へ延長でき、新規に学ぶのは Three.js の API のみ。バンドル・依存の増分が最小。ドメインカラーリングの「まずシェーダーで」判断枠組みにより、WebGPU 導入(Tier 3b)を本当に必要な場合に限定できる。

**代償・注意**:
- 命令型シーン構築はコード量が増える。共通パターン(カメラ+ライト+リサイズ+破棄)は最初の単元で確立し、以降の単元で再利用する(rule of three 原則)。
- headless e2e は WebGL1/SwiftShader であり、実 GPU との描画差(アンチエイリアス等)は検証外。数学的検証は観察表(DOM)の実値と RTL が担う。
- `preserveDrawingBuffer: true` はわずかなメモリ増。教材規模では無視できるが、大量パーティクル単元を作る際は再評価する。
