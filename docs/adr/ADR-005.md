# ADR-005: Tier 3a(標準3D)の導入方式 — vanilla Three.js + useEffect ライフサイクル

- **ステータス**: Accepted(制作者=HUMAN の「Tier1→Tier2→Tier3 へ GO」指示 2026-07-13 を Go とし、本 ADR の PR マージをもって発効)
- **決定者**: ARCHITECT(Claude Code オーケストレータ)起票 → 技術スパイク(別エージェント)→ 独立レビュー(別系統)+ 制作者事後監査
- **日付**: 2026-07-14(JST)
- **関連文書**: ADR-001(Tier 3a=Three.js は決定済み——本 ADR は**導入方式**を決める)、ADR-004(Pixi の useEffect ライフサイクル前例・a11y 条件)、`docs/DESIGN.md` §レンダリング戦略、`AGENTS.md` §3(C-3/C-4/C-6)

## 文脈

MVP 2 完了(公開29単元)により MVP 3(Tier 3: 一次変換3×3・回転行列と基底変換・曲面プロット・複素関数ドメインカラーリング)が対象になった。レンダラー自体は ADR-001 で Tier 3a=Three.js(既定 WebGL レンダラー)と決定済みだが、(a) React Island との統合方式、(b) headless e2e での検証可否、(c) バンドル影響、(d) キーボード操作(a11y DoD は削減対象外)の方式が未決だった。技術スパイク(2026-07-14、ソース: `docs/adr/assets/adr-005-three-spike/`)で vanilla 統合と @react-three/fiber(R3F)を実装比較した。

## 計測・検証(スパイク要旨)

| 観点 | 案A: vanilla + useEffect | 案B: R3F |
|---|---|---|
| コード量(同一デモ) | 146行 | 105行+`extend()`/型拡張の付帯コード |
| 追加依存 | `three` のみ(OrbitControls 同梱) | `three` + `@react-three/fiber` |
| バンドル(gzip 実測・**再現手順と生ログは assets/bundle/**) | **135.1 KB**(three+OrbitControls。独立再計測 133.7 KB) | +約103.5 KB(R3F ランタイム純増。独立再計測 +101.7 KB——~102±2 KB) |
| Astro Islands(client:visible) | 問題なし | 今回の版の組合せでは SSR/hydration 問題は再現せず |
| キーボード代替の実装難度 | 自前実装必須 | **同一**(OrbitControls はオービットのキー操作を持たず、R3F の宣言性の恩恵が及ばない) |
| headless e2e | 描画成功 | 描画成功 |

- headless Chromium(SwiftShader)は **WebGL1 のみ**だが、Three.js が自動フォールバックし、Tier 3a の標準用途(立方体・格子・矢印・OrbitControls)に WebGL2 は不要と確認。
- **e2e の非空ピクセル判定には `preserveDrawingBuffer: true` が必須**(既定 false では readPixels が常に0を返す——実装ノート)。
- 性能下見: 102 オブジェクト+OrbitControls、SwiftShader(ソフトウェア描画)でも 64〜66fps。実 GPU では十分な余裕。

## 決定

1. **Tier 3a の統合方式として vanilla Three.js(useEffect ライフサイクル管理)を採用する。R3F は採用しない**(将来、シーンの複雑化で命令型が不利になった場合の再評価候補として温存)。根拠は複合: (a) **依存最小化**(three 本体のみ。R3F は +~102KB gzip の純増〔再現手順・生ログ収録済み〕に加えメジャー更新の追随対象が1つ増える)、(b) **Pixi と同型の既承認前例**(ADR-004、`CltScene.tsx`)でライフサイクル規律・レビュー観点を再利用できる、(c) 最大の期待だった宣言性の便益が、本プロジェクトで最も手間のかかる要件(a11y のキーボード代替)に**及ばない**ことがスパイクで確認された。なお R3F の本来の主便益(props→シーングラフの宣言的同期)は、本プロジェクトの Scene が「少数の状態→少数のオブジェクト更新」に留まる設計(数学は lib/math、UI 状態は Experiment に集約)では発揮の場が小さい——シーンが React 状態と密結合する規模になれば再評価する。
2. **Scene 配置は `src/components/scenes/three/`**(§6 予約済み)。Pixi と同じ規律: mount で init・unmount で dispose・StrictMode 二重起動ガード・**WebGL 初期化失敗時は fail-with-message**(C-3。Tier 3a は WebGL なので事実上全ブラウザで動くが防御)。`lib/math` は Three.js を import しない(行列・ベクトル演算は自前の純粋 TS——Three.js の Matrix3/Vector3 を数学モデルに使わない。描画層でのみ変換する)。実装コストは小さい: 2×2 の前例(`linearTransformation.ts`)があり、3×3 の積・行列式・逆行列は同型の拡張で、不変条件テストの型もそのまま使える。なおスパイクの `matrix3.ts` は描画層ヘルパーとして Three の型を使っており、本番の lib/math 境界の見本ではない(スパイク資産の注記)。
3. **カメラ操作の a11y**: OrbitControls(マウス/タッチ)に加え、**離散的なカメラ回転ボタン(方位角/仰角を15°刻み等)+視点リセット**をキーボード操作可能な DOM ボタンとして必ず併設する。観察表(DOM)が値の主担保である原則は Tier 2 と同じ(ADR-004 の a11y 方針〔決定5と代償・注意〕および CltScene の実装慣行を継承)。canvas は aria-hidden。
4. **e2e 実装規約**: Three.js Scene は**本番含め常時** `preserveDrawingBuffer: true` で生成する(テスト時のみ切り替える分岐は「テストが本番と別物を検証する」危険の方が大きい)。教材規模(単一シーン・数百オブジェクト)では追加メモリ(バックバッファ保持、antialias 併用時はその分も)と present コストは無視できる。大量パーティクル系の単元を作る際に再評価する。RTL は Scene をスタブ(Pixi 前例)。**描画幾何(射影・変換の整合)は RTL で不可視のため、レビュー観点に「座標変換の単一定義」を必須で含める**(PR #38 の教訓の Tier 3 版)。
5. **Tier 3b(WebGPU)の判断枠組み**: MVP 3 の「複素関数ドメインカラーリング」は、まず **Three.js の ShaderMaterial(WebGL フラグメントシェーダー)**で実装を試みる。これは ADR-001/DESIGN の Tier 表で同題材が Tier 3b の**例示**に置かれていることと矛盾しない——ADR-004 のレビューで確認された通り「例示は候補であり義務ではない」(実際フーリエ級数も Tier 2 例示から計測で Tier 1 に確定した)。本 ADR はこの例示を「**まず 3a(ShaderMaterial)で試行し、下記ゲートで立たない場合の受け皿が 3b**」と運用上再解釈する。
   - **Tier 3b への昇格ゲート(客観基準、ADR-004 §3 と同水準)**: 代表対話操作(パラメータ c のドラッグ相当の連続変更)中に **平均 ≥58fps かつ p95 ≤20ms** を 640×480 以上で満たせない、**または** WebGL の GLSL ES 制約(ループ上限・precision)により必要な反復回数・数値精度で**正しい絵が出ない**(参照実装〔CPU/Canvas2D で低速に描いた基準画像〕との画素比較で系統的アーティファクトが確認される)——のいずれかが実測で示された場合のみ 3b へ。
   - **3b の実装形**(素の WebGPU/WGSL か Three.js WebGPURenderer か)はその時点で別途評価する(ADR-001 が WebGPURenderer を将来評価候補としている扱いを維持——本 ADR は Tier 3a に WebGPU を入れない決定のみ行い、3b の実装形は閉じない)。
   - **headless e2e の限界**: CI/ローカルの headless は WebGL1(SwiftShader)のみのため、シェーダー品質・性能の判定は **実 GPU(headed または実機)での計測**を根拠とする。e2e はクラッシュ・非空ピクセル・DOM 値の検証に限定する。
6. 依存追加: `three`(MIT)+ `@types/three`。バージョンは package.json で固定し、更新は依存審査に従う(本 ADR が審査記録を兼ねる)。バンドル +135KB gzip は Astro Islands の遅延読込により該当ページのみに局所化する(Pixi 前例で検証済みのメカニズム)。

## 検討した代替案

| 案 | 長所 | 短所 | 採否 |
|---|---|---|---|
| **vanilla Three.js + useEffect(本決定)** | 追加ランタイム依存ゼロ・Pixi 前例と同型・~103KB 軽い | 命令型のシーン構築コード | **採用** |
| @react-three/fiber | 宣言的 JSX、React 状態との自然な同期 | +~102KB gzip(実測2回)。OrbitControls 等は `extend()` の手作業が残る。a11y 要件に宣言性の恩恵が及ばず、本プロジェクトの Scene 規模では状態同期の便益も小さい | 不採用(将来の再評価候補) |
| @react-three/drei(R3F 補助) | OrbitControls 等の既製 JSX 部品 | R3F 採用が前提(上記に依存追加が積み増し)。キーボードオービットは drei にも無い | 不採用(R3F と同運命) |
| OrbitControls の a11y fork/拡張 | 統合方式に依らず使える | fork の保守責任を負う。離散ボタン方式(本決定 §3)の方が実装が小さく、観察表主担保の方針とも一貫 | 不採用 |
| Babylon.js | 統合的なエンジン機能 | ADR-001 の決定(Three.js)を覆す理由がない。バンドルも大きい | 不採用 |
| WebGPU(Three.js WebGPURenderer 含む) | 将来性 | Tier 3a の互換要件(主要モダンブラウザ全て=WebGL)に反する。ADR-001 で Tier 3b 限定と決定済み | 不採用(Tier 3b 判断は決定 §5 の枠組みで) |
| 3D を自前 SVG 投影で描く(Tier 1 拡張) | 依存追加なし | カメラ操作・奥行き・遮蔽の自前実装は Three.js 採用より高コスト。ADR-001 で退けた方向 | 不採用 |

## 結果

**利点**: Tier 2(Pixi)で確立したライフサイクル・a11y・e2e の規律をそのまま Tier 3a へ延長でき、新規に学ぶのは Three.js の API のみ。バンドル・依存の増分が最小。ドメインカラーリングの「まずシェーダーで」判断枠組みにより、WebGPU 導入(Tier 3b)を本当に必要な場合に限定できる。

**代償・注意**:
- 命令型シーン構築はコード量が増える。共通パターン(カメラ+ライト+リサイズ+破棄)は最初の単元で確立し、以降の単元で再利用する(rule of three 原則)。
- headless e2e は WebGL1/SwiftShader であり、実 GPU との描画差(アンチエイリアス等)は検証外。数学的検証は観察表(DOM)の実値と RTL が担う。
- `preserveDrawingBuffer: true` はわずかなメモリ増。教材規模では無視できるが、大量パーティクル単元を作る際は再評価する。
