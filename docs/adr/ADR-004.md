# ADR-004: Tier 2(高密度2D)レンダラーの計測にもとづく選定

- **ステータス**: Accepted(制作者=HUMAN の「Tier1→Tier2→Tier3 へ GO」指示 2026-07-13 を Go とし、本 ADR の PR マージをもって発効)
- **決定者**: ARCHITECT(Claude Code オーケストレータ)起票 → 計測スパイク(別エージェント)→ 独立レビュー(別系統)+ 制作者事後監査
- **日付**: 2026-07-13(JST)
- **関連文書**: `docs/DESIGN.md` §レンダリング戦略、ADR-001(Tier 階層の確定)、ADR-003(MVP 2 の着手ゲート=Tier 導入 ADR + 計測 + 人間 Go)、`AGENTS.md` §3(C-3/C-4/C-6)

## 文脈

M8 波の完了により Tier 1(Mafs/SVG)のバックログは全消化され(公開25単元)、MVP 2(高密度2D: フーリエ級数・大数の法則/中心極限定理・フラクタル)が次の対象になった。DESIGN はTier 2 の技術を「**計測後に** Pixi.js(中規模)または WebGPU compute(規模・クオリティ最優先時)」と定めており、選定には実測が必要である。同時に DESIGN の Tier 表は Tier 2 の対象ブラウザを「**主要モダンブラウザ全て**」と定めている(WebGPU 必須・フォールバックなしが許されるのは Tier 3b のみ)。

## 計測(2026-07-13 実施)

MVP 2 の実負荷を模した2シナリオ+参考1点を、同一シミュレーションロジック(`sim.js` を共有)の SVG 版 / Pixi.js v8(WebGL)版で比較した。手法・生データ・再現用ソース一式は `docs/adr/assets/adr-004-bench/` に収録。

- **環境**: Apple M4(10コア)/ macOS 26.5.2 / Chromium 149(Playwright headless)+ `--use-angle=metal` 等で実 GPU ラスタライズを使用(`gpu-check.html` で ANGLE Metal Renderer: Apple M4 を確認)。各条件2回計測の良い方。外部負荷 load average ≈ 8〜10。
- **計測方法**: rAF ベースで 10 秒実行(先頭2秒ウォームアップ除外)、平均 FPS と p95 フレーム時間。合格基準: 平均 ≥58fps かつ p95 ≤20ms。破綻: 30fps 未満。
- **シナリオ A(エピサイクル型)**: N 本の矢印連鎖が毎フレーム回転+軌跡パス(フーリエ級数の負荷形状)。
- **シナリオ B(大量試行型)**: 一様分布12個の和(CLT)を毎フレーム試行し N 個の独立点を更新(大数の法則/CLT の負荷形状)。

| シナリオ | N | SVG | Pixi.js (WebGL) |
|---|---:|---|---|
| A(矢印連鎖) | 50〜**800** | 143.6fps / p95 8.5ms — **全条件 PASS** | 143.6fps / p95 8.5ms — PASS |
| B(独立点群) | 1,000 | 143.6fps — PASS | 143.6fps — PASS |
| B | 5,000 | 138.3fps — PASS(劣化開始) | 143.6fps — PASS |
| B | 20,000 | **40.4fps / p95 29.3ms — 60fps 明確に割れ** | 143.6fps — PASS |
| B | 50,000 | **14.8fps / p95 96.8ms — 破綻** | 143.6fps / p95 8.5ms — PASS |

- **参考**: Mandelbrot 1回描画(640×480・最大反復100・純 JS + Canvas2D putImageData)= **約12.3ms**。60fps 予算(16.6ms)に対しギリギリで、連続ズーム中の毎フレーム全再計算は非現実的だが「操作確定時に再計算」なら十分速い。

## 決定

1. **Tier 2 の既定レンダラーとして Pixi.js(WebGL、v8 系、MIT ライセンス)を採用する。WebGPU compute は Tier 2 では採用しない**(Tier 3b 専用のまま)。根拠: (a) Tier 2 の互換要件「主要モダンブラウザ全て」と WebGPU の普及状況(Safari/Firefox で発展途上)は両立しない——これは計測以前の要件制約。(b) 計測上も Pixi/WebGL は本プロジェクトの最大想定負荷(独立更新 50,000 点)で性能余裕を示し、WebGPU compute を要する規模には遠く及ばない。
2. **ピクセル単位計算(フラクタルのエスケープタイム描画等)は Canvas2D(putImageData、必要なら Web Worker でオフスレッド化)を Tier 2 の補助手段として用いる**。スプライト/ジオメトリ描画である Pixi の適性外であり、Canvas2D は全ブラウザ互換を保つ。描画方式は「操作(ズーム/パラメータ)確定時に再計算」を既定とし、毎フレーム全再計算を設計に組み込まない(参考計測より)。
3. **Tier 1 → Tier 2 昇格の計測基準を定める**: 「毎フレーム独立に更新される表示要素数」が **約5,000 を超える設計の単元は Tier 2**(SVG は 5,000 で劣化開始、20,000 で 60fps 割れ、50,000 で破綻)。少数要素×高頻度更新(エピサイクル型、〜800本)は SVG が無傷なので **Tier 1 のまま**。
4. 上記基準の即時適用: **フーリエ級数単元は Tier 1(Mafs/SVG)で実装する**(計測が「Tier 2 不要」を示した——「MVP 2 の題材だから Tier 2」ではなく負荷実測で決める)。**大数の法則/中心極限定理単元は Tier 2(Pixi.js)**(数万試行の点群が主役であり SVG の破綻域)。**フラクタル単元は Canvas2D+確定時再計算**(必要に応じ Worker)。
5. アーキテクチャ不変条件の維持: `lib/math` は Pixi/Canvas を import しない(数学モデルと描画の分離)。Pixi Scene は `src/components/scenes/pixi/` に置く(§6 の予約済みディレクトリ)。Pixi 導入は依存追加として DEVELOPMENT.md §4 の審査対象(本 ADR がその記録を兼ねる)。JS 無効時も本文と数式が読める状態を保つ(C-3。Canvas/Pixi はマウントまで描画しないため noscript 文で補う——Tier 1 と同じ運用)。

## 検討した代替案

| 案 | 長所 | 短所 | 採否 |
|---|---|---|---|
| **Pixi.js(WebGL)+ ピクセル計算のみ Canvas2D(本決定)** | 全ブラウザ互換を維持。実測で最大想定負荷に余裕。コミュニティ実績・MIT | WebGL 抽象化レイヤの学習コスト。バンドルサイズ増(Island 単位の遅延読込で局所化) | **採用** |
| WebGPU compute を Tier 2 に導入 | 規模の上限が最も高い。Tier 3b と技術共通化 | **Tier 2 の互換要件(主要モダンブラウザ全て)に反する**。計測上も必要規模に達しない。C-3 のメッセージ表示運用を 2D 単元へ拡大してしまう | 不採用(Tier 3b 専用を維持) |
| SVG(Tier 1)のまま頑張る | 依存追加なし | 実測で 20,000 点 40fps・50,000 点 15fps と破綻。LLN/CLT の中核体験(大量試行)が成立しない | 不採用 |
| 素の Canvas2D で全部書く | 依存追加なし、互換性最良 | 数万点のスプライト管理・シーングラフ・ヒット判定を自作することになり、実装・保守コストが Pixi 採用を上回る。ピクセル計算用途に限れば最適なので**補助手段として部分採用** | 部分採用 |
| Three.js(WebGL)を 2D に流用 | 既に Tier 3a で採用予定 | 2D 教材にカメラ・シーングラフは過剰。DESIGN が明示的に退けた方向(2D 概念に 3D 基盤を使わない) | 不採用 |

## 結果

**利点**: Tier 2 の技術選定が実測に裏付けられ、単元ごとの Tier 判定(昇格閾値 ~5,000 要素)が再現可能な基準になった。フーリエ級数を Tier 1 に留めることで、MVP 2 で新基盤に触れる単元が LLN/CLT(Pixi)とフラクタル(Canvas2D)の2つに絞られ、リスクが局所化される。

**代償・注意**:
- 計測環境は Apple M4 1台であり、低スペック端末では SVG の破綻閾値がさらに低い(=Tier 2 判定が保守側に倒れる方向であり、閾値の妥当性は損なわれない)。表示 FPS の上限 143.6 は計測環境の合成上限であり、比較指標としてのみ意味を持つ。
- Pixi.js のメジャーバージョン更新(v8→)は破壊的変更を含みうるため、採用バージョンを package.json で固定し、更新は通常の依存審査に従う。
- ヒットテスト・キーボード操作・スクリーンリーダー対応は SVG より手数がかかる(Canvas 系は DOM を持たない)。**a11y の DoD(axe Critical/Serious 0・キーボード操作・値のテキスト表現)は Tier 2 でも削減対象外**(C-4)——観察表・ステータステキストなど DOM 側の並走表現で担保する設計を必須とする。
