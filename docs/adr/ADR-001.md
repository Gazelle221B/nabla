# ADR-001: 図解レンダリング基盤の技術選定(Tier 3a=Three.js / Tier 3b=WebGPU)

- **ステータス**: Accepted
- **決定者**: HUMAN(プロダクトオーナー)
- **日付**: 2026-07-02
- **関連文書**: `docs/DESIGN.md` §レンダリング戦略、`docs/REQUIREMENTS.md` §制約・前提、`docs/PROJECT_PLAN.md` §4

## 文脈

nablaの中心的な差別化要素は、3Blue1Brown的な視覚的直観を、動画ではなくブラウザ内のインタラクティブな図解として実現することにある。この図解が扱う数学的対象は、中学の平面図形から大学教養の3次元線形変換・ベクトル場まで幅が広く、単一のレンダリング技術では表現要件と制作コストの両方を満たせない。

本プロジェクトの企画段階で、2本の独立したAI壁打ちセッション(`meeting.md`, `meeting2.md`)が異なる結論に達した:

- 一方は「Mafs/SVGを既定とし、要素数や更新頻度が問題化した場合にのみPixi.js・Three.jsへ計測駆動で昇格する」という段階的方針(以下「案A」)。
- 他方は、参考実装 [`gpu-bullet-hell-shrine`](https://github.com/Gazelle221B/gpu-bullet-hell-shrine)(WGSL computeシェーダー+ゼロコピー描画+Rust/WASMで数千オブジェクトを120FPS描画する弾幕ゲーム実装)を根拠に、「WebGPUを図解レンダリング基盤全体の必須技術とし、非対応環境へのフォールバックは実装しない(クオリティ優先)」という方針(以下「案B」、当初のADR-001)。

これら2つの方針は、対象ブラウザ要件(§非機能要件)と正面から衝突していた。統合作業(`docs/PROJECT_PLAN.md`)で両論併記した後、ユーザーから以下の指示を段階的に受けた:

1. 「レンダリングは両方採用で適所適材的に」→ Tier 1(2D軽量・Mafs/SVG)/ Tier 2(2D高密度・Pixi.js/WebGPU)/ Tier 3(本質的3D・WebGPU)の3階層モデルへ統合。
2. 「Three.jsも適材適所で使うよ」→ Tier 3を単純にWebGPU一本化するのではなく、Three.js(標準3Dシーン)とWebGPU(シェーダー品質・大量計算)へさらに分割。

## 決定

図解レンダリング基盤を**4階層**とする。判断順序は常に軽量な技術から重い技術へ(Tier 1 → 計測 → Tier 2 → Tier 3a → 不足時Tier 3b)。

| 階層 | 対象 | 技術 | 対象ブラウザ |
|---|---|---|---|
| Tier 1: 2D軽量(既定) | 中高範囲の大半(関数グラフ、ベクトル、平面図形、確率の単純試行) | Mafs(既定)/ SVG + D3(scales・shapes・zoomのみ) | 主要モダンブラウザ全て |
| Tier 2: 2D高密度 | フーリエ級数の大量矢印、確率の大量試行、セルオートマトン、フラクタル | 計測後にPixi.jsまたはWebGPU compute(規模・クオリティ最優先時) | 主要モダンブラウザ全て |
| Tier 3a: 標準3Dシーン | 3×3線形変換、回転行列・基底変換、平面と直線の交差、球面幾何、通常の曲面プロット | **Three.js**(既定のWebGLレンダラー) | 主要モダンブラウザ全て(WebGL) |
| Tier 3b: シェーダー品質・大量計算3D | 密なベクトル場、複素関数のドメインカラーリング、大量パーティクル、加算グローが本質的な表現 | **WebGPU**(WGSL compute + ゼロコピー描画、`gpu-bullet-hell-shrine`の流儀を流用)。重い数値計算はRust/WASM候補 | WebGPU対応ブラウザのみ。非対応時は「対応ブラウザで開いてください」を表示(フォールバック実装なし) |

判断基準:
- 2×2行列やMVP 1相当の2D概念にThree.js/WebGPUを使わない(カメラ・遠近法・奥行き操作が数学的構造をかえって見えにくくするため)。
- Tier 3内では「3D=WebGPU」と決め打ちせず、まずTier 3a(Three.jsの既製シーングラフ・OrbitControls・標準ジオメトリ)で表現できるかを検討する。密な計算やシェーダー品質のグロー表現が本質的に必要な場合のみTier 3bへ進む。
- Three.js自体のWebGPURenderer採用は将来の評価候補とし、当面は既定のWebGLレンダラーで開始する(計測してから昇格する原則を踏襲)。

## 結果

**利点**: Tier 3a(Three.js)により、対象ブラウザ全体への互換性を保ったまま標準的な3D表現(3b1bの立体シーンに近い見た目)を低コストで実現できる。Tier 3b(WebGPU)は真にシェーダー品質・大量計算が必要な場面に限定投入されるため、表現クオリティと開発コストの両立が図れる。「3D=WebGPU」への安易な決め打ちを禁止したことで、React未経験・週10時間という開発体制(`docs/REQUIREMENTS.md` §制約・前提)との緊張を緩和した。

**代償**: レンダリング技術が4階層(実質5技術: Mafs/SVG/Pixi.js/Three.js/WebGPU)に増え、`components/scenes/`配下の構成がやや複雑になる。Tier 3bのみ対象ブラウザが限定される非対称性が残る(意図的な選択であり、教材の大部分がTier 1〜3aで完結するため実害は小さいと判断)。

**不可逆性に関する注記**: Tier境界(どの表現をどのTierに割り当てるか)の変更、およびTier 3bのグレースフルフォールバック追加は、本ADRの改訂+人間承認を要する(`AGENTS.md` §3 C-6)。エージェントが「親切心」で勝手にフォールバックを追加してはならない。
