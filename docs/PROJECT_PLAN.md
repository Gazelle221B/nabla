## 0. このドキュメントについて

2つの会話ログはどちらも同じ構想「日本の学習指導要領に沿って、中学数学〜大学教養数学を、動画ではなくインタラクティブな図解で学べるGitHub Pagesサイトを作る」に行き着くが、**出発点と結論の一部が異なる**。

| | スレッドA (`meeting.md`) | スレッドB (`meeting2.md`) |
|---|---|---|
| 出発点 | 「3b1bのWeb版を作りたい」から直接技術提案 | 「インタラクティブ数学サイト、ある?」という類似サービス調査から着地 |
| 到達点 | **要件定義書 v1.0 Final**まで完了。実装計画(Step 0〜5)まで確定 | **ADR-001(WebGPU必須化)**まで確定。要件定義書は着手直前で停止 |
| 固有の成果物 | PLAN.md v1.0 / 要件定義書 v1.0 Final / OSS再利用方針 / 実装計画書 | conclaveガバナンスフレームワークへの落とし込み / ADR-001 |
| レンダリング方針 | **Mafs/SVGを既定、Pixi.js・Three.jsは計測して必要になってから** | **WebGPU必須、非対応環境はフォールバックなし(メッセージのみ)** |

この2つは当初**両立しない技術方針**を含んでいたが、ユーザー判断により「両方採用し適材適所で使い分ける」ハイブリッド方針で解消済み(§4)。本ドキュメントでは以下の方針で統合する。

1. 両スレッド共通の合意事項は「確定事項」として一本化する。
2. レンダリング技術の対立は、Tier分割(§4)によって両方を採用する形で解消した。
3. スレッドAの方が要件定義・実装計画として完成度が高いため、**全体の構造・運用ルールのベースラインはスレッドAを採用**し、スレッドBのWebGPU方針はTier 3b(シェーダー品質・大量計算3D表現)に限定して統合した。スレッドAが元々Tier 3候補としていたThree.jsはTier 3a(標準3Dシーン)として復元し、3DはまずThree.jsで足りるかを検討してからWebGPUへ進む(後日ユーザーが「Three.jsも適材適所で使う」と明示指示)。

## 1. プロジェクト概要(共通合意)

- **名称(仮称)**: ナブラ / nabla(∇)
- **タグライン**: 「予想して、動かして、発見する。」
- **コンセプト**: 3Blue1Brownのような「見て・触って理解する」数学教材を、動画ではなくブラウザ上のインタラクティブな図解(+テキスト)で実現する。
- **対象範囲**: 日本の学習指導要領に準拠した中学数学 〜 高校数学(数I〜III・A〜C) 〜 大学教養数学(微積分・線形代数・確率統計)を縦に一気通貫。
- **デプロイ先**: GitHub Pages(静的ホスティング固定。サーバーサイド処理は不可)。
- **対象ユーザー**: 独学者(1人称の学習体験)。PC中心、日本語のみ、ログイン・進捗保存機能なし。
- **開発体制**: 制作者1名 + AIエージェント(Claude Code / Codex / Antigravity / Cursor)。React未経験、週10時間が上限。

### 差別化の核(黄金パターン)

Mathigon・Immersive Math・Seeing Theory・BetterExplained・Nicky Case/Bret Victorの explorable explanations 系サービスを比較調査し、以下の型に集約した:

```
問い → 予想する → 操作する → 予想との矛盾を発見する
     → 平易な言葉で説明する → 数式として形式化する
     → 別の問題へ転用する → サンドボックスで探索する
```

**操作前に必ず予想させる(認知的な賭け)**のが、単なる「触れるグラフ」との差別化ポイント。

## 3. 確定事項(両スレッド共通のベースライン)

### 3.1 技術スタック

| レイヤー | 選定 | 理由 |
|---|---|---|
| サイト基盤 | **Astro** + MDX + React Islands | コンテンツ中心、必要な箇所だけJSを配信する島(Islands)構造がGitHub Pages静的配信と相性が良い |
| 数式 | **KaTeX**(ビルド時HTML化) | MathJaxより高速。JS本体を送らずCSS/フォントのみで表示可能 |
| UIフレームワーク | **React に統一**(混在させない) | 状態管理・テスト・アクセシビリティ実装の複雑化を回避 |
| デプロイ | GitHub Actions → GitHub Pages | push毎に自動ビルド&デプロイ。サブパス配信(`/repo/`)前提で `site`/`base` を要設定 |

### 3.2 教材構成の核

- **図の三分類**: 説明図(原則静的、2〜5枚)/ ガイド付き実験(制約された操作、1〜2個)/ サンドボックス(自由操作、1個)。中核インタラクションは1〜2個に限定。
- **Island境界**: 本文と説明図は静的MDX。状態を共有する一連の実験は単一のReact Island(`InteractiveExperiment`)にまとめる。サンドボックスは別Island。Islandを細分化しすぎない。
- **ADEPT法**: 類推(Analogy)→図(Diagram)→例(Example)→平易な言葉(Plain English)→形式的定義(Technical)を説明部分に適用。

### 3.3 数学モデルと描画の分離(最重要設計)

`lib/math/` は **React・Three.js・Pixi.jsを一切importしない純粋TypeScript**とする。数値計算と描画コンポーネントを完全に分離し、同じ数学モデルをSVG版・Pixi版・テストコードから使い回せるようにする。

```
Lesson
 ├─ Prompt        問い・予想
 ├─ Scene
 │   ├─ Model       数学的状態と計算(lib/math)
 │   ├─ Renderer    Mafs / SVG / (Pixi / Three)
 │   ├─ Controls    Slider / Input / Drag
 │   └─ Narration   現在の状態を文章化
 ├─ Checkpoint     理解確認
 └─ Sandbox        自由探索
```

固有値計算の三分割: `computeEigenSystem(matrix)`(数学的結果)/ `classifyEigenSystem(result)`(教材上の状態分類)/ `stabilizeEigenvectorDirection(current, previous)`(表示上の符号連続性)。**数学的真実と表示上の便宜を混同しない**。

### 3.4 アクセシビリティ(削減不可)

- WCAG 2.2 Level AA のうち適用可能な達成基準を目標(「準拠を保証」ではなく「目標とする」)。
- **ドラッグを唯一の入力にしない**: スライダー / 数値入力 / 矢印キー / リセットボタン / 現在値のテキスト表示を必ず併設。
- Canvasには文章または表形式の代替。キーボード操作可能。`prefers-reduced-motion` を尊重しアニメーション停止可能。
- axe による自動検査で **Critical / Serious 0件**、Lighthouse Accessibility **原則100**をリリースブロッカーとする。Performance 90以上は目標値(ブロッカーではない)。

### 3.5 コンテンツ構造・カリキュラム準拠

- 分野別フォルダ構成: `geometry/ algebra/ calculus/ linear-algebra/ probability/ discrete-math/`(学年フォルダではなく分野別。難易度・前提単元は frontmatter で管理)。
- 中学校=学習指導要領(2017告示)、高等学校=同(2018告示)準拠。大学教養は全国統一カリキュラムが存在しないため、複数大学の初年次シラバスを比較した**独自参照カリキュラム**を定義。
- frontmatter は `type: mext`(中高)/ `type: independent`(大学教養)で判別する構造にする。
- 前提知識グラフの**自動生成マップは記事20〜30本後**に先送り(メタデータの記録は今から、可視化は後から)。

### 3.6 検証体制

- MVP 1の成功指標は「閲覧」ではなく「理解」: 操作前に説明できなかった数学的関係を、操作後に説明できるようになったか。ただし**被験者が確保できていないため、教育効果はMVP 1では未検証として公開**する(「教育効果を検証済みの教材」ではなく「インタラクティブ教材の設計仮説を実装した公開ベータ」と表示)。
- AIレビューの役割分担: **Claude Code**=実装・コードレビュー / **Codex**=テスト生成・境界値・反例探索 / **別の汎用モデル**=数学的説明と学習順序のレビュー / **制作者**=最終責任者。
- **AIレビューの限界を明記**: AIは「もっともらしく見えて誤った数学的説明」を自ら生成しうるため、AIレビューは不変条件テスト等の機械的検証の**補助**であり、代替ではない。(実際、要件定義書自身に「単位行列・回転行列を特異行列と誤分類する」というAIの誤りが混入し、レビューで訂正された実例がある)
- `lib/math` には入出力テストだけでなく**不変条件ベースのテスト**を必須にする(ランダム入力での不変量検証。平行移動・回転・スケール不変性など)。自己確認的なテスト(`c = Math.hypot(a,b)` を作った式に戻すだけ等)は禁止。

### 3.7 ライセンス

| 対象 | ライセンス |
|---|---|
| ソースコード(`src/components/`, `src/lib/`, `scripts/`) | GPL-3.0-or-later |
| MDX記事・文章・独自図版(`src/content/`, `public/lesson-images/`) | CC BY-SA 4.0 |
| 第三者素材・フォント・依存ライブラリ | 各素材固有のライセンス |

- Build in public(公開リポジトリで開発)。将来非公開へ移行する可能性はあるが、**既に公開・配布した版のライセンスと利用権は撤回できない**。
- MVP期間中は**外部コードPRを原則受け付けない**(Issue・フィードバックは歓迎)— 将来のクローズ化の余地を残すため。
- リポジトリ非公開化時もGitHub Pagesサイトを維持する場合、GPLv3のソース提供義務との整合のため、対応ソースへのリンクをフッターに残す運用ルールを設ける。
- 独自ドメインは使わず `*.github.io` のまま。正式公開前に商標検索を行う(現時点は仮称)。

### 3.8 スコープ管理

- **Vision**(最終目標・現要件の対象外): 中学〜大学教養数学の網羅。
- **MVP 1**(具体的な実装対象): 三平方の定理(中学) / 微分係数と接線(高校) / 2×2行列と固有ベクトル(大学教養)の3記事を、**Mafs/SVGのみ**で作る。
- **Current scope**: 常に着手中の3〜5記事だけを具体的な約束とする。「いつか網羅する」と「今作る」を混同しない。
- **MVP 1 非対象**: ログイン/アカウント、バックエンドAPI、学習進捗管理、状態保存・共有、コメント機能、CMS、多言語対応、自動採点式演習、検索機能、前提知識グラフ自動生成、Pixi.js / Three.js、ライトモード。
- 期間: 最初の1記事 4〜6週間、MVP 1全体 **12週間目標・上限16週間**。超過時は期間延長ではなく機能削減(削減順: 装飾アニメーション → 実例1件のみの早すぎる汎用化 → GA4カスタムイベント → 付加的な説明図 → 記事数を3本→2本へ。**サンドボックス・予想・数学テスト・アクセシビリティは削減対象外**)。

## 4. 確定: レンダリング戦略(ハイブリッド)

スレッドA(段階的にMafs/SVG→Pixi→Three.js)とスレッドB(WebGPU必須・ADR-001)の対立は、**「両方採用し、適材適所で使い分ける」**というユーザー判断で解消した。全面WebGPU化(案Bそのまま)でも全面Mafs/SVG化(案Aそのまま)でもなく、**表現要件に応じて技術層を積み上げる4階層モデル**(Tier 1・2・3a・3b)を確定版とする。当初はTier 3をWebGPU一本に統合していたが、ユーザーから「Three.jsも適材適所で使う」との追加指示があり、Tier 3自体もThree.js(標準3D)とWebGPU(シェーダー品質・大量計算)へさらに使い分けるよう改訂した——スレッドAが元々Tier 3候補としていたThree.jsを、単純にWebGPUへ置き換えるのではなく併存させる形。

### 4階層レンダラー選定基準

| 階層 | 対象 | 技術 | 対応MVP |
|---|---|---|---|
| **Tier 1: 2D軽量**(既定・全記事のベースライン) | 中高範囲の大半(関数グラフ、ベクトル、平面図形、確率の単純試行) | **Mafs**(既定)/ SVG + D3(scales・shapes・zoomのみ) | MVP 1〜 |
| **Tier 2: 2D高密度**(要素数・更新頻度が問題化した場合) | フーリエ級数の大量矢印、確率の大量試行、セルオートマトン、フラクタル | 計測後に **Pixi.js**(中規模)または **WebGPU compute**(規模・クオリティを最優先する場合) | MVP 2〜 |
| **Tier 3a: 標準3Dシーン** | 3×3線形変換、回転行列・基底変換、平面と直線の交差、球面幾何、通常の曲面プロット — シーングラフ・カメラ操作(OrbitControls)・標準ジオメトリで表現できるもの | **Three.js**(既定のWebGLレンダラー) | MVP 3〜 |
| **Tier 3b: シェーダー品質・大量計算3D**(3b1b的なグロー表現が本質的に必要) | 密なベクトル場、複素関数のドメインカラーリング(ピクセル単位計算)、大量パーティクル | **WebGPU**(WGSL compute + ゼロコピー描画 + 加算グロー、`gpu-bullet-hell-shrine`の流儀を流用)。重い数値計算はRust/WASMを候補とする | MVP 3〜 |

判断順序は常に **Tier 1 → 計測 → Tier 2 → (Tier 3aで検討 → 不足時Tier 3bへ)**。「2×2行列やMVP1相当の2D概念にThree.js/WebGPUを使わない」という案Aの原則は維持する(カメラ・遠近法・奥行き操作が数学的構造をかえって見えにくくするため)。Tier 3内でも「安易に3D=WebGPU」と決め打ちせず、まずThree.jsの既製部品(シーングラフ・OrbitControls・標準ジオメトリ)で足りるかを検討し、密な計算やシェーダー品質のグロー表現が本質的に必要な場合のみTier 3bへ進む。

### フォールバック方針(スコープを分離して両立させる)

ADR-001の「WebGPU必須・フォールバックなし」という結論は、**Tier 3b(シェーダー品質・大量計算3D)に限定して**維持する。これにより、サイト全体の対象ブラウザ要件(§3.1・§3.4)とWebGPU必須方針は衝突しなくなる。

- **Tier 1・Tier 2・Tier 3a**: 従来通り主要モダンブラウザ全て(Chrome/Edge/Firefox最新2バージョン、macOS Safari現行、iOS Safari/Android Chrome現行)で動作することを維持。Tier 3a(標準3DシーンはThree.js/WebGL)も対象ブラウザ全てで動作するため、MVP 1〜3aの記事は全読者がアクセスできる。
- **Tier 3bのみ**: WebGPU非対応環境へのグレースフルフォールバックは実装しない(クオリティ優先)。非対応ブラウザでは無言でクラッシュさせず、「対応ブラウザ(Chrome/Edge等)で開いてください」という明示メッセージのみ表示する。これはページ単位・コンポーネント単位のスコープであり、サイト全体をWebGPU必須にはしない。
- 結果として、独学者は**Tier 1〜3aの記事(標準的な3D空間表現を含む)は環境を問わず読める**。Tier 3bの記事(密なベクトル場・ドメインカラーリング等、シェーダー品質が本質的なごく一部の単元)のみ、対応ブラウザが必要になる。

### ADR-001(改訂版): スコープをTier 3b(WebGPU)とTier 3a(Three.js)に分離する

> スレッドB原案(全面WebGPU必須)を**上書き・改訂**。元の決定内容(WGSL compute・ゼロコピー描画・Rust/WASM候補・フォールバックなし・非対応時メッセージ表示)自体は維持するが、**適用範囲を「図解レンダリング基盤全体」から「本質的にシェーダー品質・大量計算が必要なTier 3bコンテンツ」へ縮小する**。さらに、スレッドAが元々Tier 3候補としていたThree.jsを、単純にWebGPUへ置き換えるのではなくTier 3a(標準3Dシーン)として復元する(ユーザー指示: 「Three.jsも適材適所で使う」)。
>
> - 決定: Tier 3b(密なベクトル場・複素関数ドメインカラーリング・大量パーティクル等、シェーダー品質と大量計算が本質的なもの)のレンダリングにWebGPU(WGSL compute + ゼロコピー描画)を必須採用する。Tier 3a(3×3変換・曲面・勾配・回転行列・基底変換・球面幾何等、シーングラフとカメラ操作で表現できるもの)はThree.jsを用いる。Tier 1・2はMafs/SVG/Pixi.jsを用い、いずれもWebGPUを要求しない。
> - Tier 3内の判断順序: まずTier 3a(Three.js)で表現できるかを検討し、密な計算やシェーダー品質のグロー表現が本質的に必要な場合のみTier 3b(WebGPU)へ進む。「3D=WebGPU」と決め打ちしない。
> - Tier 3bコンポーネント内でのみ、非対応環境はフォールバックせず「対応ブラウザで開いてください」のメッセージ表示に留める。Tier 3a(Three.js/WebGL)は対象ブラウザ全てで動作するためこの制約を受けない。
> - Rust/WASM(Vite + wasm-pack)はTier 3bの重い数値計算が必要になった時点で候補評価する(§5のOSS再利用方針と同じ「事前一括採用しない」原則に従う)。
> - この改訂も含め、Tier境界の変更は本ADRの再改訂 + 人間承認を要する。

## 5. OSS再利用方針(車輪の再発明の回避)

スレッドAでは、要件定義後に「車輪の再発明になっていないか」を自己点検し、既存の教材プラットフォーム(Mathigon Studio, PreTeXt, Idyll)を調査した。

### 採用判断

| 対象 | 判断 | 理由 |
|---|---|---|
| Astro + MDX + React Islands | 維持 | — |
| Mafs | MVP 1標準レンダラーとして採用 | — |
| Mathigon Studio | **不採用**(設計・教材表現の参考資料に留める) | Node.js/Expressサーバー前提でGitHub Pages(静的)要件と根本的に非整合。採用は「部品の再利用」ではなく「別プラットフォームへの移住」になる |
| PreTeXt | **不採用**(アクセシビリティ設計の参考資料) | XML著作モデルへの移行+独自React体験の接合コストが、既存基盤を使う利点を上回る |
| Idyll | **不採用**(思想・記事構成の参考資料) | マルチページ運用が手動、開発が停滞気味でエコシステムが小さい |
| `@mathigon/euclid`(MIT) | 候補(事前非採用) | 三平方の直角制約はMafsの`useMovablePoint`の制約関数で実装可能。同一幾何処理が複数箇所で重複するなど基準を超えたら評価 |
| `@mathigon/fermat`(MIT) | 候補(事前非採用) | 2×2固有系の教材用分類は汎用行列ライブラリより解析式の自作の方が挙動・テスト・教材説明を一致させやすい |
| Mathigon教材コンテンツ | **再利用しない** | `mathigon/textbooks`リポジトリは `UNLICENSED` かつ `private: true`。「GitHubで読める=再利用可能」ではない |

### 「車輪の再発明」の定義

- **避けるべき自作**: 汎用座標系、ドラッグ/ズーム/パン、数式レンダリング、一般的な幾何交差アルゴリズム、汎用行列演算、CMS、独自Markdownパーサー、独自アクセシビリティ基盤。
- **再発明ではない自作(ナブラのドメインロジック)**: 三平方の残差を求める関数、接線の傾きの教材固有モデル、2×2固有系の教材用分類、予想と結果を対応付ける状態遷移、誤解を顕在化させるガイド付き操作、数学的状態を自然言語に変換する処理。

依存ライブラリは事前に一括採用せず、記事実装中に具体的な必要性が発生した時点で追加し、重要な依存追加はADRに記録する。

### Step 0〜5(M0〜M1の作業分解)

1. **Step 0: 要件凍結** — 要件定義書v1.0 Final、PLAN.md v1.0、OSS再利用方針、`MATH_CONVENTIONS.md`、`LICENSES.md`、`DEVELOPMENT.md` をファイル化。以降の変更はIssue/ADR経由のみ。

4. **Step 3: 最小React実験** — 予想 / 制約付き可動点 / 代替入力 / 残差表示 / リセット / キーボード操作。ドラッグと数値入力を同一React状態に同期。
5. **Step 4: 記事として閉じる** — 平易な説明 / 形式化 / 代表的誤解 / 転用問題 / 最小サンドボックス / JS無効時の代替文章。
6. **Step 5: 品質ゲート** — Unit test / 型チェック / build / GitHub Pagesサブパス動作 / Playwrightスモークテスト / axe(Critical/Serious 0件) / 手動キーボード試験 / モバイル幅確認。

### 最小垂直スライスの定義(重要な設計是正)

当初「最初の成果物 = `lib/math/pythagoras.ts`」としていたが、これでは以下が検証できないと指摘され是正された:

> React Islandの境界が適切か / Mafsと数学モデルが分離できるか / ドラッグと代替入力が同期するか / MDX本文と実験が自然につながるか / GitHub Pagesのサブパスで動くか / キーボード・タッチ操作が成立するか / JS無効時のフォールバックが読めるか

これらは**縦に貫通させて初めて検証できる**ため、最初の成果物は「数学モデル→Mafs表示→代替操作→静的説明→デプロイを一度貫通した最小教材」と再定義された(walking skeleton)。

### MVP 1 完了条件(Definition of Done)の要旨

- **教材**: 学習目標1〜2個に限定 / 操作前に予想を要求 / 代表的誤解1つ以上 / 平易な説明→形式的定義 / 転用問題またはサンドボックスで終わる。
- **操作**: マウス・タッチ可 / ドラッグ以外の代替入力あり / キーボードのみで主要操作完了可 / リセット可 / 不正値・特異値・境界値で破綻しない / モバイル幅で意味が失われない。
- **技術**: 数学モデルの単体テスト / 不変条件テスト / コンソール未処理例外0 / サブパスで動作 / JS無効でも本文と数式は読める / axe Critical・Serious 0件 / Lighthouse Accessibility原則100。
