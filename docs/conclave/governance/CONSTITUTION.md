# Pillar 2 — The Constitution: 絶対 NG (不変条件)

> **憲法は全エージェントの全行動に優先する。** 他のあらゆる指示・最適化・善意より上位。
> いずれかに抵触しそうな場合は**実装を止めて人間へエスカレーション** ([ESCALATION.md](ESCALATION.md))。

---

## なぜ「やってはいけないこと」を列挙するのか

AI は善意で機能を足し、コードを「改善」し、スコープを広げ、便利のために制約を回避する。これを止めるのは能力ではなく**明文化されたルール**である。憲法は「やれること」ではなく「やってはいけないこと」を列挙する — 禁止は許可より誤解されにくく、違反が検出しやすい。

憲法は `AGENTS.md` の最上位セクション (§絶対NG) に要旨を置き、全文を本ファイル相当に展開する。300 行制限の `AGENTS.md` には**要旨と禁止事項だけ**を残し、詳細は各設計ドキュメントへポインタを置く。

## 分類のテンプレート (C-1〜C-8)

自プロジェクトの憲法を以下の枠で起草する。各項目は「検出可能」な粒度で書く (誰が見ても違反が判定できる)。C-1〜C-6 は抽出元由来の汎用枠、C-7/C-8 は v0.2 で研究知見から追加。

### C-1. リポジトリ運用 NG

- [ ] **保護ブランチ (`main` 等) への直接 push 禁止。** 実装は必ず `agent/<task-id>-impl` ブランチ。
- [ ] **merge は人間承認のみ。** AI は merge 不可 (例外は明示的人間許可のみ)。
- [ ] **hook スキップ禁止** (`--no-verify` / `--no-gpg-sign` 等)。
- [ ] **秘密ファイル (`.env` 等) の commit 禁止。** `.env.example` のみ管理。
- [ ] **PR マージ後はブランチを使い回さず、最新の保護ブランチから切り直す** (コンフリクト防止)。

### C-2. データ・スキーマ NG

- [ ] 一意制約の設計を勝手に変えない (例: `UNIQUE(<複合キー>)` を単一カラムに張り替えない)。
- [ ] 空キー・必須欠落レコードの INSERT 禁止。書き込み直前にアサート。
- [ ] ID/キーの**生成順を変えない** (再現性・冪等性の前提)。
- [ ] エンコーディング単位を固定 (例: バイト単位の文字列切り詰め禁止、コードポイント単位で扱う)。

### C-3. 耐障害性 NG

- [ ] **1 単位 (1 ソース/1 タスク) の失敗で全体を止めてはならない** (fail-open)。
- [ ] **副系 (通知・配信) の失敗で主系 (収集・保存) を fail させてはならない。** ログ記録のみ。
- [ ] **タイムアウト未指定の外部呼び出し禁止。** リトライ回数の上限を定める。

### C-4. スコープ境界 NG (フェーズ境界)

- [ ] 現フェーズの DoD を満たすまで、**次フェーズの機能を導入しない** (スコープ膨張防止)。
- [ ] コスト上限を超える呼び出し方の変更は**人間判断** (勝手に上限を上げない)。
- [ ] 「ついでに〜したい」と思ったら即停止し、`PROJECT_STATE` の「人間判断待ち」へエスカレーション。

### C-5. コンテンツ・法務・倫理 NG (プロジェクト固有)

- [ ] `<プロジェクト固有の禁止表現・転載・権利・コンプライアンス>` を記入。
- 抽出元の例: 記事本文の無断転載禁止 (要約のみ)、無断声クローン禁止、特定ナラティブ/政治断定の禁止、AI 生成物の明示義務。

### C-6. 技術スタック NG

- [ ] **確定済みの言語/構成の決定 (ADR) を勝手に覆さない。** 変更は ADR 追記 + 人間承認。
- 抽出元の例: 「単一言語へ統一」の決定を多言語ハイブリッドへ戻さない。

### C-7. 終了 NG (termination) — v0.2 新設

> 根拠: MAST (arXiv:2503.13657) で「終了条件の無認識」(12.4%) + 「早すぎる終了」(6.2%) ≒ **全失敗の 19%** が終了に関する失敗。Conclave のフェーズ・レベルの人間ゲートだけでは per-task の終了を縛れていなかった盲点。

- [ ] **全ての委任タスクは、明示的な完了/終了条件と最大反復上限を持たねばならない。**
- [ ] エージェントは**定義された出口なしにループしてはならない**(無限ループ・トークン溶解の防止)。
- [ ] エージェントは**終了条件を満たす前に「完了」と宣言してはならない**(早すぎる終了の防止)。
- RUNBOOK の決定木は委任の前に「このタスクは書かれた終了条件 + 反復上限を持つか?」を確認する。

### C-8. 過剰修正 NG (unverifiable ≠ false) — v0.2 新設

> ⚠️ **Conclave 由来の設計判断**(出典の Hallucination Cascade はこの規則を勧めていない。同論文の実測は「修正パスのうち約 3.8% が元々正しかった内容を誤って書き換える / 各 hop で事実性が 0.789→0.769 へ低下」という trade-off まで)。Conclave はこの実測を受けて、独自に不変条件化する。

- [ ] **検証/修正パスは、確認できないという理由だけで先行エージェントの事実主張を削除・書換してはならない。** 確認不能 (unverifiable) は偽 (false) ではない。
- [ ] 確認できないクレームは**フラグして escalate**(ARCHITECT か人間へ、区分 B/C)するのであって、下流パスが黙って破壊しない。
- 詳細は [JUDGE_RELIABILITY](../principles/JUDGE_RELIABILITY.md) JR-5。

## 改訂プロトコル

憲法は軽々に変えない。変えるときは:

1. 重大な設計判断は **ADR (Architecture Decision Record)** を追加する。
2. `AGENTS.md` の §絶対NG / §アーキテクチャ方針 / §ドキュメント地図を同期更新する。
3. `PROJECT_STATE.md` の「直近の設計判断」に記録する。

> 憲法と他のルールが矛盾したら、**常に憲法を優先**する。Karpathy 4 原則 ([EXECUTION_DISCIPLINE](../principles/EXECUTION_DISCIPLINE.md)) ですら、憲法と衝突する場合は憲法が勝つ。
