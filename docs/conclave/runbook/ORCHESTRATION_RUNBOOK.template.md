# 自律オーケストレーション運用書 (ORCHESTRATION_RUNBOOK) — テンプレート

> **対象読者: 本プロジェクトのオーケストレーターを務める AI エージェント (またはその後継)。**
> 目的: オーケストレーターが交代しても、この 1 冊を読めば「現在地の判定 → 次の一手の決定 → 適切な AI への委任 → 検証 → 記録」を**人間の逐次指示なしに**回せるようにする。
> 位置づけ: `ROLE_TOPOLOGY` が *組織契約* (誰が何の役割か) を定義するのに対し、本書は *操作手順* (どう自走するか) を定義する。両者は補完関係。
> **不変条件 (CONSTITUTION の絶対 NG) は本書より常に優先。** 矛盾したら憲法に従う。
>
> 🔧 `<...>` のプレースホルダを自プロジェクトの値で埋めて使う。本書は陳腐化しない**恒久手順**に保ち、時点情報は `HANDOFF.md` / `PROJECT_STATE.md` に隔離する。

## 1. 起動シーケンス (段階的コールドスタート)

> 文脈窓は有限資源(注意の予算)。全ドキュメント先読みは予算を浪費する。**EAGER な最小コアだけ即時に読み、残りは随時 (on-demand) 軽量参照から読む**(→ [Context Hygiene](../principles/CONTEXT_HYGIENE.md) CH-5)。それでも「とりあえず実装」は始めない。

**EAGER (即時・最小コア — 必ず先に読む):**
1. `AGENTS.md` — 最上位の指示書・禁止事項 (§絶対NG)。
2. `docs/PROJECT_STATE.md` の「現在のフェーズ」と「次に実行すべきアクション」「人間判断待ち」 — **真の記憶**。内部記憶より常にこちらを信じる。
3. 本書 §2 の決定木 — 次の一手の決め方。

**ON-DEMAND (随時・軽量参照から必要時に読む):**
4. `docs/TEST_LOG.md` の末尾 — 直近の運用実走の結果(運用判断のときに)。
5. `docs/HANDOFF.md` — 直近の引き継ぎ時点スナップショット(交代直後に)。
6. `docs/DESIGN.md` の該当節 — 実装/レビュー対象が確定してから、その節だけ。

読んだら **§2 の決定木で現在地を判定**してから動く。

## 2. 状態判定 → 次アクションの決定木 (本書の核)

`PROJECT_STATE.md` の「現在のフェーズ」と git/PR 状態を突き合わせ、**上から順に最初に当てはまる行**を実行する。

| # | 判定条件 (確認コマンド) | 現在地 | 次の一手 |
|---|---|---|---|
| 1 | `git status` が dirty | 作業途中で中断 | 未コミット変更の意図を PROJECT_STATE と照合 → §4 品質ゲート → commit (作業ブランチへ)。意図不明な変更は破棄せず人間判断待ちに記録 |
| 2 | `<テスト/ビルドコマンド>` が赤 | 品質ゲート破れ | §4 を緑にするまで他作業を止める。原因が自分の変更でなければ `git log` で特定し PROJECT_STATE に記録 |
| 3 | レビュー待ちの PR がある | レビュー段階 | `REVIEWER` スロットに独立レビューを委任 (§3)。実装者には投げない |
| 4 | レビュー PASS かつ QA 未実施 | QA 段階 | `QA_MEMORY` スロットに受け入れ検証を委任 (§3) |
| 5 | レビュー+QA PASS かつ完了 PR 未作成 | 完了直前 | DoD 更新 → 完了 PR 作成 (`gh pr create`)。**merge はしない** (§5 人間ゲート) |
| 6 | 完了 PR マージ済み かつ 次フェーズ Go 未宣言 | 次フェーズ判断待ち | **停止して人間を待つ** (§5)。判断材料を `PROJECT_STATE`「人間判断待ち」に準備済みにする |
| 7 | 人間が次フェーズ Go を PROJECT_STATE に記録済み | 次フェーズ着手可 | 最新の保護ブランチから `agent/<next-task>-impl` を切り、計画書の次タスクから §3 委任サイクルで実装 |
| 8 | 上記いずれも非該当 | 平常運用 | §6 日次/定常運用ループ + ドキュメントのドリフト点検 |

> **原則**: 決定木が「人間ゲート」を指したら、追加作業を発明せず停止する。ゲート迂回はプロダクトを劣化させる。

### 委任前の 2 ゲート (v0.2 — 実装着手の前に必ず)

決定木が「実装へ進む」を指したら、`IMPLEMENTER` に投げる**前に**この 2 つを通す:

1. **ANALYZE — 文書整合ゲート** (spec-kit `/analyze` の Conclave 版): `REQUIREMENTS ↔ DESIGN ↔ IMPLEMENTATION_PLAN` が相互に矛盾せず、全受け入れ条件が計画タスクで被覆されているかを検証する。**必ず独立スロット (`REVIEWER` または `QA_MEMORY`、起草した `ARCHITECT` ではない)** が行う(自己分析にしない=P1 視点独立)。ギャップがあれば実装を止める。コード検証 (P6) は fresh に行うのに、最も高価なサイクルの前に文書整合は検証していなかった盲点を塞ぐ。
2. **終了条件チェック** (憲法 C-7): その委任タスクは**書かれた完了/終了条件と最大反復上限**を持つか? 無ければ先に定義する。出口なしのループと早すぎる終了を防ぐ。

## 3. 外部 AI ルーティング (誰に投げるか・正確な呼び出し方)

> **大原則**: オーケストレーター (あなた) のトークンは希少資源。他の AI ができることは委任し、自分は**計画・統合・判断・検証**に集中する。
> **acting 委任は SSOT を運ぶ**: `IMPLEMENTER`/`ARCHITECT` への委任は「タスク + 完了条件 + 触らない範囲」だけでなく、**依存する上流決定 (DESIGN 該当節 + PROJECT_STATE の制約) を含めるか指し示す**。スコープ一行だけは矛盾決定を招く ([ROLE_TOPOLOGY](../roles/ROLE_TOPOLOGY.md) §委任の鉄則 1)。
> **戻りは蒸留 + 検証**: 委任先は生トランスクリプトでなく蒸留要約 (1–2k トークン) + 参照を返す。採用前に参照の根拠を自分で検証する ([Context Hygiene](../principles/CONTEXT_HYGIENE.md) CH-4)。記録したら生出力を窓から落とす (CH-2)。
> **検証必須(コードも判定も)**: 委任結果は必ず自分で検証してから採用する。コードは §4 ゲート、出典 URL は実在確認。**委任された判定 (review/QA verdict) も検証する**——引用 file:line の実在・引用テストの fresh 通過(判定者は証拠を幻覚する → [JUDGE_RELIABILITY](../principles/JUDGE_RELIABILITY.md) JR-7)。無検証で「事実」として記録しない。
> **役割分離**: 実装者・レビュアー・QA は必ず別 AI。PR 作者 ≠ レビュアー。judging スロットは生成元と別系統 (self-enhancement bias)。同一コンテキストで自己承認しない。
> **ドメイン別の検証強度**: 幻覚リスクは話題依存(抽象・弱接地の主題ほど高い)。抽象的・事実主張の多い出力には深いクレーム分解検証 + 追加の修正 hop を、テストが通る決定的なコードには軽い検証を割り当て、希少な検証労力を配分する。

| 用途 | スロット | 非対話呼び出し | モデル指針 / fallback |
|---|---|---|---|
| 日常実装・リファクタ・テスト | `IMPLEMENTER` | `<impl-cli> run -m <model> "<prompt>"` | 主: `<低コストモデル>`。**上限到達時** fallback: `<無料/予備モデル>` |
| 独立レビュー (実装に関与させない) | `REVIEWER` | `<review-cli> exec "<prompt>"` | `prompts/review.md` を使用。判定は PASS/FAIL + 重大度別件数 |
| 最終 QA・整合性確認 | `QA_MEMORY` | `<qa-cli> -p "<prompt>"` | `prompts/qa.md`。受け入れ条件チェック + ドリフト検出 |
| 調査・セカンドオピニオン | (検索グラウンディング) | `<search-cli> -p "<prompt>"` | 一次情報 URL は**必ず自分で実在検証** |
| Git/PR ワークフローのみ | (Git 専用) | `gh pr create` / `gh pr view` | **コーディングには使わない** (リクエスト上限) |

### Peer messaging transport (任意: agmsg)

複数の CLI エージェントが**同時に**動くプロジェクトでは、`agmsg` をローカル peer messaging transport として使ってよい。`agmsg` は `bash` + `sqlite3` のローカル SQLite 経由で Claude Code / Codex / Gemini CLI / Copilot CLI / Antigravity などの peer session をつなぐ。Conclave の役割は変えない。

利用判断:
- 使う: 複数の live agent がいて、人間の copy-paste 仲介を減らしたい。
- 使わない: 単独セッション、単発作業、永続記録だけで十分な作業。

導入例(人間またはプロジェクト owner が明示的に選ぶ):
```bash
npx agmsg
# or
npm i -g agmsg && agmsg install
```

ガードレール:
- `agmsg` は transport。**PROJECT_STATE / HANDOFF / REVIEW_REPORT / QA_REPORT / GitHub PR を置き換えない**。
- `agmsg` で合意した判断・ブロッカー・検証結果は、採用前に必ず repo 内の該当 SSOT へ転記する。
- 実装者からレビュアーへの直接メッセージは許可されるが、レビュー独立性と fresh verification は緩めない。
- 秘密値・認証トークン・顧客データを送らない。
- `agmsg` が壊れた場合は環境失敗として扱い、2 回連続失敗で区分 D エスカレーションする。Conclave の本線は止めない。

委任の型 (推奨):
```bash
# 実装委任 — バックグラウンド実行して結果を回収
<impl-cli> run -m <model> "<具体的タスク + 完了条件 + 触ってはいけない範囲>" > /tmp/oc-out.txt 2>&1 &

# 独立レビュー
<review-cli> exec "$(cat prompts/review.md) 対象: <Ticket>。基準: DESIGN.md / IMPLEMENTATION_PLAN.md"

# QA
<qa-cli> -p "$(cat prompts/qa.md) 対象: <Ticket>。DoD: IMPLEMENTATION_PLAN.md の該当節"
```

> **委任先の既知の不調と対処をここに記録する** (例: バージョン X 以下のバグ、upgrade で解消、2 回連続失敗で区分 D として人間へ)。陳腐化しやすいので確認コマンド付きで残す。

## 4. 品質ゲート (完了宣言前に必ず・記憶で代用しない)

```bash
<テストコマンド>     # 全テスト緑
<lint コマンド>      # lint クリーン
<型チェックコマンド> # 型クリーン
```

3 つとも**フレッシュ実行で**緑を確認してからのみ「完了」と言う。秘密値 (API キー / Webhook / トークン) をログ・ドキュメント・コミットに含めない。
