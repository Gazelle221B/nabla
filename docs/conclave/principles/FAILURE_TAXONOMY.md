# Pillar 6+ — Failure Taxonomy: 実証された失敗モード × Conclave の防御

> Conclave の設計直観を、**実証データに接地**させる。
> 出典: Cemri et al.「Why Do Multi-Agent LLM Systems Fail?」(MAST), arXiv:2503.13657 — 7 つの主要マルチエージェント基盤、1,600+ の注釈付き実行トレース、アノテーター間一致 κ=0.88 に基づく初の大規模実証研究。
> 中核命題: **マルチエージェント失敗の大半は、単一モデルの能力不足ではなく組織・設計の欠陥である**(「洗練された個人の組織でも、組織構造が欠陥なら破滅的に失敗しうる」)。これは Conclave が立脚する前提そのもの。

## MAST 14 失敗モード × 防御クロスウォーク

頻度は MAST の実測値。各モードを防ぐ Conclave の機構を対応づけ、**未カバーのモードを正直に明示**する。

### FC1 — システム設計の問題 (約 44%)

| FM | 失敗モード | 頻度 | Conclave の防御 |
|---|---|---|---|
| 1.1 | タスク仕様違反 | 11.8% | REQUIREMENTS の受け入れ条件 + DESIGN §7 + 高レベル DoD 検証 (P6) |
| 1.2 | 役割仕様違反 | 1.5% | 役割スロットの明示定義 (P1)。**低頻度=スロット定義が効いている実証** |
| 1.3 | ステップ反復 | **15.7%** | `PROJECT_STATE` による状態永続化 (P3) + context-reset cadence (CH-6) |
| 1.4 | 会話履歴の喪失 | 2.8% | 永続化 > 内部記憶 (P3) |
| 1.5 | 終了条件の無認識 | **12.4%** | ⚠️ **新設 C-7 終了 NG**(従来未カバー)→ [CONSTITUTION](../governance/CONSTITUTION.md) |

### FC2 — エージェント間の不整合 (約 31%)

| FM | 失敗モード | 頻度 | Conclave の防御 |
|---|---|---|---|
| 2.1 | 会話リセット | 2.2% | 永続化 (P3) |
| 2.2 | 明確化を求めない | 6.8% | ⚠️ **新設・先行的 clarify ゲート**(従来は Class C で事後対応のみ)→ [ESCALATION](../governance/ESCALATION.md) |
| 2.3 | タスク脱線 | 7.4% | 憲法 C-4 スコープ境界 + 「ついで」禁止 (P2) |
| 2.4 | 情報の出し惜しみ | 0.85% | acting ロールへ SSOT を渡す規則 (P1/P4) |
| 2.5 | 他エージェント入力の無視 | 1.9% | DESIGN を唯一の真実の源として参照 (P3) |
| 2.6 | 推論-行動の不一致 | **13.2%** | ⚠️ **新設・推論-行動整合チェック**(差分が宣言した計画と一致するか)→ [ROLE_TOPOLOGY](../roles/ROLE_TOPOLOGY.md) §委任の鉄則 |

### FC3 — タスク検証 (約 23%)

| FM | 失敗モード | 頻度 | Conclave の防御 |
|---|---|---|---|
| 3.1 | 早すぎる終了 | 6.2% | 高レベル DoD 検証 + 終了条件 C-7 |

## この表が教える 3 つのこと

1. **既存設計の実証的裏付け**: ステップ反復 (15.7%) + 履歴喪失 (2.8%) + 会話リセット (2.2%) ≒ 全失敗の **21% が揮発性状態の失敗**。これは P3「persistence > internal memory」がまさに防ぐもの——直観が数字で裏付けられた。

2. **3 つの盲点(v0.2 で塞いだ)**:
   - **終了条件** (FM-1.5 12.4% + FM-3.1 6.2% ≒ 19%) → 憲法 C-7 + 決定木の per-task 終了チェック。
   - **推論-行動の不一致** (FM-2.6 13.2%、単一最頻の不整合モード) → 役割分離だけでは捕捉不能。差分 vs 宣言計画の照合が要る。
   - **検証者の信頼性** (FM-3.3 誤検証 9.1%、FC3 最頻) → Conclave は REVIEWER/QA を品質の最後の砦にするが、検証者自身が最大級の失敗源。→ JUDGE_RELIABILITY で対処。

3. **高レベル検証が最高レバレッジ**: MAST の最強の介入結果は「低レベル正しさチェックの上に**高レベルの目的検証**を足す」ことで **+15.6%**。次いで**役割仕様の改善で +9.4%**。
   → Conclave の DoD を**2層**(低レベル: テスト/lint/型が緑 / 高レベル: 成果物が REQUIREMENTS の目的を満たすか)に分割する根拠([EXECUTION_DISCIPLINE](EXECUTION_DISCIPLINE.md))。
   → 同時に、役割仕様の改善も大きな実測ゲイン (+9.4%) を生むため、**役割の明示は「過剰設計」ではない**(役割違反が 1.5% と低いのは仕様が効いている証拠)。

## 正直な限界 (intellectual honesty)

MAST は「**文脈・通信プロトコルに焦点を当てた解決策は、エージェント間不整合 (FC2) にしばしば不十分**で、より深いアーキテクチャ変更が要る」と警告する。Conclave のハンドオフ機構 (P3: ドキュメントを唯一の工程間チャネルとする) は構造化通信プロトコル解の一種であり、この警告の射程内にある。

→ Conclave の立場: ドキュメント・ハンドオフは FC2 を**低減するが消去しない**。残余リスクは「視点の独立性 (judging ロール) + 二段検証 + verify-before-adopt」で抑える——プロトコル単独で解けるとは主張しない。この限界を明示することが、限界を隠すより信頼に足る。

> 自己監査の使い方: 新規プロジェクトはこの表を**チェックリスト**として使い、「各失敗モードに対し、自分のプロジェクトの防御は何か」を埋める。空欄が盲点である。
