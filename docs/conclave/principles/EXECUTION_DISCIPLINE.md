# Pillar 6 — The Discipline: 実行規律

> Andrej Karpathy の LLM コーディング 4 原則を、**検証可能な手続き**へ翻訳する。
> 出典: [multica-ai/andrej-karpathy-skills](https://github.com/multica-ai/andrej-karpathy-skills)。
> **憲法 (CONSTITUTION) と矛盾する場合は憲法を優先**する。

---

## 1. Think Before Coding — 実装前に思考する

**勝手に仮定しない。混乱を隠さない。トレードオフを提示する。**

手続き化:
- **設計書 (DESIGN.md) を読まずに実装着手しない。** 設計矛盾を発見したら実装を止め、`PROJECT_STATE`「人間判断待ち」へ (区分 B)。
- 複数解釈があれば**全部出す**。例: 取得失敗時に「例外を raise」/「`Result.error` に包む」のどちらか — 後者が fail-open に合致するが、根拠を ADR か実装コメントに残す。
- **曖昧なら先に問う(先行的 clarify)。** 不明点は**着手前に一括で列挙**し `[NEEDS CLARIFICATION: <質問>]` として人間へ 1 パスで渡す。これは「2 回詰まる」より前の第一級トリガー(MAST「明確化を求めない」6.8% への先行対処 → [ESCALATION](../governance/ESCALATION.md))。「レビューで初出の疑問」は要件失敗 (区分 C) の兆候。

## 2. Simplicity First — 最小実装

**問題を解く最小コード。投機的な機能は書かない。**

手続き化:
- 単一用途のための抽象化を禁止。将来使うかもしれないプロトコル/プロバイダ抽象を**先回りで書かない** (必要になった ADR の時点で導入)。
- 「ジュニアエンジニアに『これ複雑すぎ』と言われないか?」を自問し、Yes なら書き直す。
- 200 行書いて 50 行で済むなら書き直す。
- **最長の単一スレッドを優先する (don't fork without a reason)。** ある作業単位を別の acting エージェントへ分割/委任する前に、それが本当に分離を要するか確認する——分離が正当化されるのは「独立レビュー」「希少資源の集中」「文脈窓の枯渇」のいずれか。そうでなければ 1 本のスレッドに保つ。各分割は文脈断片化のコスト(Cognition の脆さ)を払う。役割分離はそのコストに見合う所(judging の独立性)にだけ置く。

## 3. Surgical Changes — 外科手術的変更

**触るのは必要な箇所だけ。自分が作ったゴミだけ片付ける。**

手続き化:
- `agent/<task-id>-impl` ブランチ内では、そのタスクに**直接トレースできない変更を禁止**。「ついでに既存の型ヒントを整理」は NG。
- 既存スタイルに合わせる (行長・型厳格度・既存設定を変えない)。
- 未関連の dead code を見つけても**削除せず**、`PROJECT_STATE`「人間判断待ち」に記載に留める。
- 自分の変更で orphan になった import/関数/変数のみ削除可。

## 4. Goal-Driven Execution — 目標駆動実行

**成功基準を定義し、検証されるまでループする。**

手続き化 — 命令形を検証可能形へ変換:

| 命令形 (弱い基準) | 検証可能形 (強い基準) |
|---|---|
| 「fail-open を実装」 | 「1 ソースが例外でも他が完走し `consecutive_failures` が +1 されるテストを緑にする」 |
| 「dedupe を入れる」 | 「同一キーを 2 回 insert すると 1 行のままになるテストを書く」 |
| 「バグを直す」 | 「バグを再現するテストを書き、それを緑にする」 |

多段タスクは `1. ... → verify: ...` 形式で `TEST_LOG.md` に記録する。

## 完了の定義 (Definition of Done)

> 明文化しない場合「一応動いた」が完了扱いされる。各段階の DoD を満たさない限り次工程に進めない。

| 段階 | 完了条件 |
|---|---|
| **実装完了** | `IMPLEMENTER` がコード変更・ローカルテスト通過・変更要約・既知制限を提出し、`TEST_LOG.md` に実行ログを保存済み |
| **レビュー合格** | `REVIEWER` が Critical/High 指摘ゼロ (Medium 以下は許容/Issue化) と判定し、`REVIEW_REPORT.md` に**証跡欄** (確認ファイル・根拠行・テスト結果) が記入済み |
| **検収可能** | `QA_MEMORY` が ① DESIGN・実装差分・テスト結果・README 更新の整合 (低レベル) と ② **成果物が REQUIREMENTS の目的を満たすか (高レベル目的検証)** の両方を確認済み |
| **完了** | 上記すべてを満たし、**人間が merge を承認**済み |

### 二段の検証 (低レベル + 高レベル) — v0.2

> 根拠: MAST の最強の介入結果は「低レベル正しさチェックの上に**高レベルの目的検証**を足す」ことで **+15.6%**(役割仕様の改善でも +9.4%)。Conclave の DoD は低レベルゲート (テスト/lint/型) に偏っていた。

完了は**2 層**で判定する:
1. **低レベルゲート** — テスト/lint/型が fresh 実行で緑(下記「フレッシュ実行の鉄則」)。「コードが動く」。
2. **高レベル目的検証** — `QA_MEMORY` が「**この成果物は REQUIREMENTS の目的を実際に満たすか**」を確認する。「動く」ではなく「正しいものを作った」。`QA_REPORT` に記録。

低レベルだけ緑でも高レベルが未充足なら完了ではない。最大のゲインはこの高レベル層にある。

### フレッシュ実行の鉄則

「完了」と言う前に、品質ゲート (テスト/lint/型) を**その場で実行**して緑を確認する。**記憶で代用しない。** 「さっき緑だった」「ローカルで動いた気がする」で止めない。3 つとも fresh 出力で緑になって初めて完了。

### 証跡なき判定を出さない

レビューは必ず**具体的な `file:line` とテスト結果**に紐付ける。根拠行が書けないなら PASS せず「確認不能」と書く。これを義務付けないと、レビューが「もっともらしい評論」になり、実際のコードを根拠としない判定が混入する。

### クレーム分解検証 — v0.2

> 根拠: Hallucination Cascade のアブレーションで、**原子的クレーム分解を外すと検出性能が最も大きく劣化**した(検証の最重要プリミティブ)。

事実を含む出力(REVIEW_REPORT の指摘、QA 散文、調査/セカンドオピニオン結果、生成コンテンツ)は、**全体一発で判定せず、原子的クレームに分解して各々を個別に grounding** する。各クレームを `PASS / UNVERIFIABLE / FAIL` で記録。これは「file:line 証跡」のクレーム粒度への一般化。詳細は [JUDGE_RELIABILITY](../principles/JUDGE_RELIABILITY.md) JR-4。

### 事実性後退ガード (no-overcorrection) — v0.2

> 根拠: Hallucination Cascade — 各修正 hop で事実性が低下し (0.789→0.769)、修正のうち**約 3.8% が元々正しかった内容を誤って書き換える** (overcorrection)。検証は「欠陥除去」だけを報酬にすると、正しい内容を巻き添えで破壊しうる。

review/QA パスが通るのは、**欠陥を除去しつつ、検証済みの正しい内容を正味で失っていない**ときのみ。確認できないクレームは**フラグ**するのであって**黙って削除しない**(unverifiable ≠ false、憲法 C-8)。「正しいクレームの過剰修正」を明示的なレビュアー失敗モードとして扱う。

> **同じ場所で 2 回以上詰まる** → 能力の限界ではなく**問題定義の欠陥**。コードではなく要件を見直す。

詳細は [governance/ESCALATION.md](../governance/ESCALATION.md)。
