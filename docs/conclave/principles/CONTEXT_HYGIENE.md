# Pillar 5+ — Context Hygiene: 文脈は有限資源

> **オーケストレーターと各エージェントの「生きたコンテキスト窓」そのものが、有限で枯渇する資源である。** ファイルへの永続化 (P3) はその裏返し——窓を守るために真実をファイルに逃がす。
> 出典: Anthropic「Effective Context Engineering for AI Agents」、Cognition「Don't Build Multi-Agents」、BMAD-METHOD。
> 位置づけ: P3 (The Record) は「真実をどこに置くか」、本書は「**生きた窓をどう節約するか**」。両者は表裏一体。

---

## なぜ独立した関心事なのか

Conclave の「希少資源の集中投下」(P1) は当初**モデルのコスト**だけを指していた。しかし Anthropic が示すのは、もう一つの別個の希少資源——**どのエージェントの文脈窓も持つ「注意の予算 (attention budget)」**——の存在である。Transformer の注意は n トークンに対し n² の関係を張るため、トークンが増えるほど recall と長距離推論が劣化する(「context rot」)。これは**閾値で崩れる崖ではなく、緩やかな勾配**である。

→ ゆえに「完了宣言前にゲートを通す」「状態をファイルに書く」だけでは不十分で、**生きた窓を能動的に節約する操作**が要る。それが Context Hygiene。

---

## CH-1. Flush-then-Compact (永続化してから圧縮する)

文脈窓が上限に近づいたら圧縮 (compaction: 会話を要約して新しい窓で再開) するが、**Conclave では順序が逆**になる:

1. **先に** 耐久状態を `PROJECT_STATE.md` / `HANDOFF.md` / ADR へ flush する。
2. **その後で** 要約して再開する。
3. 圧縮要約を、設計判断・未解決バグ・実装詳細の**唯一の住処にしてはならない**。

> Anthropic の既定は「要約して窓を作り直す」。だが圧縮要約は lossy であり、Cognition は「要約は暗黙の決定を落とす」と警告する。Conclave の永続化層は圧縮層でもあるため、**圧縮の前に必ずファイルへ確定**させる(P3 との整合)。要約は recall 最大化を先に、precision は後で調整する。

## CH-2. Tool-Result Clearing (記録済みの生出力を窓から落とす)

委任結果・ツール出力を**ファイルに記録したら、生出力を作業窓から捨てる**。生のトランスクリプトを抱え続けると注意予算を食い潰す。記録 → 破棄が基本動作。

## CH-3. Just-in-Time Loading (必要なときに読む)

全ドキュメントを先読みせず、**軽量な識別子**(ファイルパス・チケットID・ADR番号)を保持し、実行時にツールで本文を読む。メタデータ(名前・階層・日時)が「いつ何を読むべきか」のシグナルになる。

→ これは Conclave の**段階的コールドスタート読み込み**と直結する(下記 CH-5、RUNBOOK §1)。

## CH-4. Distilled Return Contract (蒸留して返す)

委任したサブエージェントは、**生のトランスクリプトではなく蒸留した要約 (目安 1,000–2,000 トークン) + 全文への参照**を返す。重い探索はサブエージェントの隔離窓で行い、オーケストレーターは高レベルの計画と統合に専念する。

> Anthropic: サブエージェントは数万トークン使っても、返すのは 1–2k の蒸留要約。**ただし Conclave の留保**: オーケストレーターは検証者でもある (P4/P6) ため、PASS を記録する前に蒸留要約の根拠 (file:line・テスト結果) を自分で確認する。蒸留と検証は両立させる——要約を信じるのではなく、要約 + 検証可能な参照を受け取る。

## CH-5. 段階的コールドスタート (tiered read)

RUNBOOK の起動シーケンスを「全部先読み」から**2層**に分ける:

- **EAGER (即時・最小コア)**: `AGENTS.md` + `PROJECT_STATE.md` の「次アクション」+ 決定木。これだけは必ず先に読む(「とりあえず実装」を防ぐ最小核)。
- **ON-DEMAND (随時)**: `TEST_LOG` 末尾・`HANDOFF`・`DESIGN` の該当節を、必要になった時点で軽量参照から読む。

> これは「先読みで注意予算を浪費する」(Anthropic) と「読まずに実装着手させない」(Conclave) の正直な折衷。Anthropic 自身がこの hybrid (一部を先読み + 残りを探索) を推奨している。

## CH-6. Context-Reset Cadence (こまめにリセットする)

**1 タスク/1 ストーリーを完了するたびに、文脈を圧縮して新しい会話を始める。** 再開の基盤はチャット履歴ではなく `PROJECT_STATE` + 当該の作業単位ドキュメント。

> BMAD: 「文脈が育つほどエージェントの性能は劣化する。こまめに、できれば各ストーリー実装後に、会話を圧縮して新しい会話を始めよ」。これは P3「persistence > internal memory」の運用カデンス版であり、P6 の fresh-verification と同じ精神。

## CH-7. 右の高度のプロンプト (right altitude)

`AGENTS.md` ・役割プロンプト・委任プロンプトを書くときの高度:

- **低すぎる**(脆いハードコード if-else ロジック)でも **高すぎる**(曖昧な一般論)でもなく、**強いヒューリスティクスの最小集合**を、明確に区切ったセクション (見出し・XML タグ) で。
- 最小から始め、観測された失敗モードに対して反復する。
- `AGENTS.md` の **≤300 行ルールは「注意予算の節約」** として理解する(可読性だけの話ではない)。

> **実例**: 本番 AI エージェントの統治文書が「右の高度」で実際どう書かれているかは、Conclave リポジトリ/配布物の `references/system-prompt-as-governance.md`(Claude Fable 5 システムプロンプト標本の読み解き)を参照——明確に区切られた節・優先順位つきの不可侵則・「前提を確かめる」検証規律の実例。パスはバージョン非依存(`references/` は配布物に同梱されるが、study material のため `conclave init` では導入先に撒かれない)。

## 適用チェックリスト

- [ ] 圧縮の前にファイルへ flush したか (CH-1)
- [ ] 記録済みの生出力を窓から落としたか (CH-2)
- [ ] 全文先読みでなく軽量参照 + 随時読み込みにしたか (CH-3, CH-5)
- [ ] 委任の戻りは蒸留要約 + 参照か(かつ採用前に検証したか)(CH-4)
- [ ] タスク完了ごとに文脈をリセットしたか (CH-6)
- [ ] プロンプトは「右の高度」で最小集合か (CH-7)

> **モデル改善の留保 (Anthropic)**: 能力の高いモデルほど prescriptive な足場を必要としない。本書の規律は「文書化された失敗モードを防ぐ最小限」に留め、モデル世代交代ごとに**下方へ**再評価する([ROLE_TOPOLOGY](../roles/ROLE_TOPOLOGY.md) の再評価条項と連動)。
