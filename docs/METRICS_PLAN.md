# METRICS_PLAN — M9a 計測基盤の合格基準・許可リスト事前登録

- ステータス: 事前登録(実装前コミット)。ADR-006 M9a の完了条件は「基準文書がマージ済み」であることを要求しており、本書がその基準文書に当たる。
- 関連: `docs/adr/ADR-006.md`(M9 全体方針)、`docs/DESIGN.md` §非機能要件(GA4 安全要件)、AGENTS.md C-5(教育効果を「検証済み」と表示しない)。
- 適用範囲: 公開中の全単元ページ(`/lessons/{slug}/`)。

## 1. 目的宣言(この計測が「証明」しないもの)

この計測の目的は **「中核ループ(予想 → 操作 → 発見)の致命的な破綻を検出すること」** であり、**教育効果の証明ではない**。

- 「破綻の検出」とは例えば「予想入力率が極端に低い」「操作実施率が極端に低い」といった、コア体験が機能していない兆候を早期に見つけることを指す。
- 被験者による事前/事後/遅延テストは実施しない(ADR-006、制作者判断)。行動テレメトリはあくまで実利用シグナルの代理指標であり、相関であって因果ではない。「操作後に読者が自分の言葉で説明できるようになったか」という北極星指標そのものを直接測るものではない。
- 本計測の結果をもって「教育効果を検証済み」と表示しない(AGENTS.md C-5、不変)。合格基準を満たしても「学習効果があった」ではなく「中核ループが機能不全ではなさそうだ」としか言えない。この区別は README・サイト上のいかなる文言にも波及させない。

## 2. 送信してよいデータの境界(許可リストの外側は送らない)

- 送信するのは **イベント名** と **単元 slug(`unit_slug`)** のみ。
- **送信しないもの(明示的禁止)**: スライダー・数値入力の値、予想の選択肢そのもの(正誤や具体的な回答内容)、自由記述、氏名・メールアドレス等の個人特定情報、Cookie を使った長期的なユーザー識別、IPアドレスの生値(GA4 既定の匿名化に委ねる)。
- **予想履歴(ADR-006 M9c、`nabla:predictions:v1`)はローカルのみ**: 単元・選んだ予想の内容・確定日時は、この端末のブラウザの `localStorage` にのみ記録し、GA4 を含めどこにも送信しない(ネットワーク送信そのものが発生しない実装)。上記「予想の選択肢そのもの」の送信禁止は GA4 カスタムイベントに対する制約だが、M9c の予想履歴機能はそもそも許可リスト(`unit_slug` のみ)を拡張しない——予想内容を運搬する経路を新設しない、という境界を明文化する。演習(ExerciseSection)の解答・正誤も同様にどこにも送信しない。
- **予想履歴の記録は1ページロードにつき最大1件**(`prediction_submit` イベントの fireOnce と同じ設計判断): 確定ボタンの連打・二重クリック等で同一内容の重複レコードが履歴に積み重ならないよう、`predictionHistoryRecorder.ts` は最初の確定操作のみを記録する(`docs/METRICS_PLAN.md` の対象は GA4 だが、同一の重複防止設計思想を共有するためここに明記する)。
- GA4 の既定自動収集(`page_view` / `session_start` 等、URL・リファラ・大まかな地域を含む)は無効化しない。これは GA4 プロパティである以上避けられない標準ベースラインであり、本書が対象とする「独自送信データ」の対象外として扱う(`docs/DESIGN.md` の「GA4は数値入力・自由記述・個人特定情報を送信しない」はカスタムイベントの送信内容に対する制約と解釈する)。Google シグナル・広告向け機能は有効化しない。IP 匿名化は GA4 の既定動作に委ねる(`anonymize_ip: true` を明示指定)。

## 3. イベント定義の許可リスト(これ以外は送らない)

| イベント名(GA4) | 日本語 | 発火条件 | 送信属性 |
|---|---|---|---|
| `prediction_start` | 予想入力開始 | 単元ページ内で予想の選択肢(ラジオボタン)を最初に選んだ時点 | `unit_slug` |
| `prediction_submit` | 予想確定 | 「予想を確定して実験する」ボタンを押した時点(以降ドラッグ・スライダー操作が可能になる) | `unit_slug` |
| `experiment_interact` | 操作(実験)実施 | 予想確定後の実験セクション内にある操作コントロールを最初に操作した時点(下記の一般化された定義) | `unit_slug` |
| `lesson_complete` | 単元完了 | §4 の操作的定義を満たした時点 | `unit_slug` |

- 各イベントは **1 ページ読み込みにつき単元ごとに最大 1 回** のみ発火する(多重カウント防止。§6 実装対応)。
- 属性は `unit_slug: string` の 1 個のみ。TypeScript の型(`NablaEventParams`)でこれ以外のプロパティを渡すとコンパイルエラーになるようにし、実装がこの許可リストから逸脱できないようにする(§6)。

### `experiment_interact` の一般化された定義(2026-07-24 改訂)

初版では「操作コントロール」を `-slider`/`-number` の id を持つ `<input>` に限定していたが、独立レビュー(Kimi K2.7)で **`GraphTheoryExperiment`(辺のON/OFFを SVG の `role="switch"` 要素で行い、`-slider`/`-number` の `<input>` を一切持たない)だけが検出漏れになる**ことが判明した。そのため「操作コントロール」を以下へ一般化する(コード側は `src/lib/analytics/coreLoopObserver.ts` の `isOperateControl` と 1:1):

> 予想確定後にのみ操作可能になる、実験セクション(`section[aria-labelledby$="-exp-title"]`)内の対話要素——`<input>` への `input`/`change`(予想ラジオを除く。プリセット選択ラジオ・スライダー・数値入力を含む)、または `button`・`[role="button"]`・`[role="switch"]`・`[tabindex="0"]` への `click`/`keydown`(Enter/Space)——への最初の操作。予想確定ボタン自身(文言「予想を確定して実験する」)は除外する。

これにより `<input>` 以外の対話パターン(SVGのスイッチ等)を持つ将来の単元にも、個別配線なしで対応できる。実験セクションの外側にある要素は対象外(スコープ制限)。

## 4. 「単元完了」の操作的定義

**静的サイト(バックエンドなし)で検証可能な定義**として、以下を採用する:

> 予想確定後にのみ出現する操作コントロール(§3 の一般化された定義)を **少なくとも1回操作した上で**、予想確定後にのみ出現する「予想と結果」(チェックポイント)セクションの見出しが **1000ms 以上継続してビューポート内に留まった** 時点を「単元完了」とする。

### 根拠

- 全単元共通の DoD 構造(`docs/DESIGN.md` 黄金パターン)は「予想 → 操作(Scene + Controls)→ 観察 → 確認(Checkpoint)」の順に一本道で進む。チェックポイントは実装上、予想確定 (`submitted === true`) の場合にのみ DOM へ現れるため、チェックポイントの出現自体が予想確定を含意する。
- しかし「チェックポイントが見えた」だけでは、スクロールで素通りしただけの離脱を「完了」と誤カウントしうる。これを避けるため「実際に操作コントロールを動かしたか」を追加条件にし、離脱直前にたまたま見えた場合のノイズを排除する。
- ドウェル時間(1000ms)はチェックポイントの文章を読み始めるのに十分な最小限の滞在時間として設定した経験的な閾値であり、GA4 の Enhanced Measurement のスクロール到達度(90%到達)のような粗い代理指標より、実際の理解確認セクションへの到達を直接指標にできる。
- ページ遷移・進捗保存を伴わない静的サイトのため、「テスト正答」のような従来型の完了定義は使えない(採点にはバックエンドか localStorage が要る — localStorage 併用の演習は M9c の管轄)。M9a は GA4 のみで完結する定義に限定する。

## 5. 合格基準の事前コミット

初回計測のため過去データがない。以下はエンジニアリング上の合理的な初期仮説であり、実測が蓄積し次第 `docs/PROJECT_STATE.md` に改訂記録を残した上で見直すことを前提とする(固定的な絶対基準ではない)。

**最小サンプルサイズ**: 観測ウィンドウ(既定 30 日ローリング)内で当該単元の `prediction_submit` が 30 件未満の場合、その単元の判定は「サンプル不足につき判定不能」とし、合否いずれの判定も下さない(小標本による誤検知を避ける)。

| 指標 | 算出式 | 破綻(FAIL)判定のしきい値 | 意味 |
|---|---|---|---|
| 予想入力率 | `prediction_start` 数 ÷ 単元ページの `page_view` 数 | 20% 未満 | ページに来ても予想UIにすら触れない = ゲート導線・可視性の問題 |
| 予想確定率 | `prediction_submit` 数 ÷ `prediction_start` 数 | 50% 未満 | 予想はするが確定まで至らない = 選択肢・ボタンの導線問題 |
| 操作実施率 | `experiment_interact` 数 ÷ `prediction_submit` 数 | 50% 未満 | 確定しても操作しない = 「動かして発見する」という核心体験が使われていない |
| 単元完了率 | `lesson_complete` 数 ÷ `prediction_submit` 数 | 30% 未満 | 操作はしても結果の確認まで到達しない = 観察・確認セクションの導線/分量の問題 |

- いずれか1指標でもしきい値を下回り、かつ最小サンプルサイズを満たす場合を「中核ループの致命的破綻」の疑いありとし、当該単元を優先調査対象として `docs/PROJECT_STATE.md` の未解決リスクへ記録する。
- 全単元がしきい値を上回ることは「効果が実証された」ことを意味しない(§1 不変)。あくまで「使われている」ことの下限確認にとどまる。

## 6. 実装への対応(許可リストの強制・非本番無効化)

- `src/lib/analytics/events.ts`: 許可リストのイベント名 union 型(`NablaEventName`)と送信属性型(`NablaEventParams = { unit_slug: string }`)を定義。これ以外のイベント名・属性は型エラーになる。
- `src/lib/analytics/ga4.ts`: `trackEvent(name, params)` が `window.gtag` の有無を確認し、無ければ何もしない安全なノーオペレーション(コンソールエラーを出さない)。
- `src/lib/analytics/coreLoopObserver.ts`: §3・§4 の発火条件を DOM イベント委譲(ラジオ `change` / ボタン `click` / 操作コントロールへの `change`・`input`・`click`・`keydown` / `IntersectionObserver`)で実装。既存の32単元 Experiment Island 自体には手を入れず(構造的規約への準拠を利用)、`unit_slug` は URL パス `/lessons/{slug}/` から導出する。
- **非本番(dev/preview)では計測無効**: GA4 スクリプトの出力は `import.meta.env.PUBLIC_GA4_ID`(未設定 = 無効)と `import.meta.env.PROD`(`astro dev` では false)の両方を満たす場合のみ行う。測定 ID は本 ADR 起票時点で未発行(HUMAN ゲート)のため、現時点では常に無効(dev・preview・CI・本番ビルドいずれも同じ挙動になる)。将来 HUMAN が GitHub Actions の `deploy.yml` にのみ `PUBLIC_GA4_ID` シークレットを注入する運用を想定しており、ローカル環境にこの値が漏れることは C-1(`.env` commit禁止)により構造的に防がれる。
  - 残存リスクとして明記: 開発者が本番用 ID を手元の `.env` に複製して `astro build && astro preview` を実行した場合、`PROD` フラグは true になるため計測が有効化されてしまう。この経路は運用規約(シークレットを手元に複製しない)で防ぐものであり、コードでは検出できない。今回のテストはこの経路を検証対象にしない。
- **構造契約の回帰検出**: `src/components/lesson/__tests__/experimentContract.test.ts` が全32単元 Experiment(+ それが import する Scene コンポーネント)のソースを走査し、実験セクション・予想ラジオ・確定ボタン文言・チェックポイント見出し・(一般化後の定義に基づく)計測可能な操作コントロールの存在を検証する。新単元の追加やリファクタで構造契約が破れた場合、無音の計測欠落ではなく CI の FAIL として即座に検知する。
